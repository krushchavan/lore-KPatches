🔥 TODAY Re: [PATCH v2 2/3] btrfs: fix incorrect buffered IO fallback for append direct writes Needs Work LLM (per-reviewer) Under Review

This patch addresses a potential issue in Btrfs where buffered IO fallback for append direct writes could lead to incorrect i_size updates, causing readers to block on an out-of-evaluation (OE) error and potentially return zeroes. The problem arises from the fact that the invalid i_size is only fixed up within the dio thread holding the inode lock, but released before the extent lock is acquired. Boris Burkov suggests this could lead to a reader observing the intermediate too-big i_size before it's corrected. The patch aims to clarify and fix this potential issue.

🔥 TODAY Re: [PATCH v2 8/9] nfsd: hold net namespace reference for delayed-dispose nfsd_files Needs Work LLM (per-reviewer) Under Review

This patch addresses a false positive concern about a potential bug in the nfsd code. The issue was that the global shrinker unhashes and isolates an nfsd_file onto a local dispose list, but then the net namespace teardown completes and struct net is freed before the shrinker resumes and calls get_net() on freed memory. However, the reviewer's analysis shows that this scenario cannot happen because the struct net outlives all pernet exit callbacks, including nfsd_net_exit(), and the isolation and get_net() happen in the same execution context. The patch does not change any code, but rather confirms that the existing placement of get_net() is correct and ensures that nf_net remains valid for an arbitrarily long time while the file sits on the per-net dispose list waiting for an nfsd thread to close it.

This patch modifies the compact_gap() function in the Linux kernel's mm/compaction module to cap its output at COMPACT_CLUSTER_MAX, rather than allowing it to grow indefinitely. The original implementation of compact_gap() was causing issues with THP (Transparent Huge Pages) allocations and reclaim operations. By capping the gap, the patch aims to reduce unnecessary reclaim and improve system performance. Initial testing showed improvements in THP allocation rates, but further investigation revealed that globally capping the gap is a better approach than capping it at specific downgrade gates. The patch's author has provided detailed test results and analysis to support this conclusion.

[PATCH 0/4] arm64: cross-CPU NMI via SDEI Positive llm-chunked-no-replies Accepted

This patch series adds a new mechanism to deliver NMI (Non-Maskable Interrupt) on arm64 CPUs, using the SDEI (System Description Event Infrastructure) firmware interface. This is necessary because current mechanisms have limitations: pseudo-NMI has a high performance cost and is disabled in some systems, while FEAT_NMI is not available in deployed silicon. The new mechanism uses event 0 of the standard SDEI software-signalled events and allows interrupting CPUs with locally masked interrupts without affecting their hot path performance. This will enable features like hard-lockup detection, all-CPU backtraces, and crash dumps to work correctly even when pseudo-NMI is disabled.

This patch adds a new function called sdei_event_signal() to the Linux kernel's firmware/arm_sdei.c file. The function signals a software-signalled event (event 0) to a target processor, making it pending and delivering an NMI-like signal even if interrupts are masked on that processor. This is achieved through a thin wrapper over the SDEI_EVENT_SIGNAL call, which takes no locks and is therefore safe to call from NMI or crash context. The patch also updates the include/linux/arm_sdei.h file to add a declaration for sdei_event_signal(), and adds a new constant to the include/uapi/linux/arm_sdei.h file to define the SDEI_EVENT_SIGNAL function number.

[PATCH 4/4] arm64: route crash_smp_send_stop() last resort through SDEI Neutral llm-chunked-no-replies Awaiting Review

This patch adds the SDEI (System Description Event Interface) as a last resort to stop CPUs in case of a crash on arm64 architecture. It first attempts to stop CPUs using normal IPI and pseudo-NMI IPI, but if those fail, it uses SDEI to send a cross-CPU NMI that can reach even wedged CPUs with interrupts masked. The SDEI handler captures the CPU's state before parking it, making it less recoverable than a normal stop. This approach is chosen because an SDEI-stopped CPU never completes its event and is thus less recoverable.

The fix involves moving the check for !CONFIG_SWAP below the early return for non-swap entries, ensuring that only true swap entries trigger warnings and non-shmem pages follow the existing 'uptodate / non-shmem' path.

This patch moves the setup of Hugepage Vmemmap (HVO) to early initialization, replacing the previous fallback mechanism that lost HVO optimization on error. The new path panics if vmemmap_populate_hvo() returns an error, which may be a stricter requirement than intended for HVO as an optimization rather than a hard necessity.
