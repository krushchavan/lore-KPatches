🔥 TODAY [PATCH 0/5] btrfs: improve stalls under sudden writeback Needs Work LLM (per-reviewer) Under Review

This patch series aims to improve stalls in the Btrfs file system under sudden writeback conditions, particularly on systems with large amounts of memory and disk space. The problem arises when a large number of dirty pages are accumulated before writeback occurs, leading to a tailspin in metadata space management. To address this, the patches introduce improved reservations for delayed refs in delalloc work, smooth out the work in flush_space(), and reduce the size of the unit of work in shrink_delalloc(). The changes aim to prevent stalls by reducing the number of ordered extents and delayed refs blocking transactions.

🔥 TODAY [PATCH 1/5] btrfs: reserve space for delayed_refs in delalloc Needs Work LLM (per-reviewer) Under Review

This patch addresses a problem where the Btrfs file system does not reserve enough space for delayed references (delayed_refs) during delalloc operations, leading to incorrect assessments of reserved metadata space and poor decision making in reclaim logic. The issue arises from the fact that the reservation is over-estimated but not meticulously tracked as each ordered extent is created. As a result, when writeback occurs and delayed refs are generated, they draw from an under-reserved trans_handle's delayed_refs_rsv. This can lead to erroneous assessments of reserved metadata space and poor decision making in reclaim logic. The patch aims to fix this by reserving space for delayed_refs in delalloc operations.

[PATCH 3/5] btrfs: account for compression in delalloc extent reservation Neutral llm-chunked-no-replies Awaiting Review

This patch addresses an issue in Btrfs where the maximum number of extents reserved for delayed refs reservations during delalloc writes was under-estimated when compression is enabled. The problem arose from using a fixed estimate that didn't account for compression, leading to potential issues with reservation and release calculations. To fix this, the patch introduces new functions to calculate the maximum extent size and max extents based on inode properties, ensuring accurate estimation even in compressed scenarios. As a result, delayed refs reservations are no longer under-reserved during delalloc writes, even when compression is forced or always enabled for an inode.

[PATCH 5/5] btrfs: cap shrink_delalloc iterations to 128M Neutral llm-chunked-no-replies Awaiting Review

This patch limits the iterations of the `shrink_delalloc` function in Btrfs to 128M, preventing it from blocking indefinitely and causing high syscall latencies. The function is responsible for reclaiming space in the file system, but its current implementation can lead to long stalls when dealing with large amounts of delalloc data. By capping the iterations at 128M, the patch aims to strike a balance between making progress on space reclaim and avoiding arbitrary blocking. This change should improve the overall performance and responsiveness of the system, especially in scenarios where writeback is heavy and async reclaim is blocked.

Ongoing Submitted 2026-03-24 🔥 TODAY [PATCH] generic/747: handle ENOSPC gracefully during write/delete cycles Needs Work LLM (per-reviewer) Changes Requested

This patch addresses a test failure issue in the generic/747 test, which consistently fails on btrfs due to ENOSPC errors during write/delete cycles. The problem arises from statfs overestimating available space at high fill levels, causing dd to hit ENOSPC even when there's room. To handle this, the patch breaks out of the fill loop when full, deletes a file to free space and retry on failure, and redirects dd stderr to preserve errors for debugging purposes. The test is designed to stress garbage collection through write/delete churn, not to validate space accounting.

This patch adds a new ioctl (input/output control) called BTRFS_IOC_GET_CSUMS, which allows userspace to query the on-disk checksums for a file without needing to read the data itself. The ioctl takes an argument specifying the offset and length of interest, as well as a buffer to store the results. The kernel returns a struct containing information about whether the checksum is available, and if so, what type it is (e.g., sparse, preallocated, or compressed). This feature is primarily intended for speeding up mkfs.btrfs --rootdir operations, particularly when using the --reflink option, which can result in significant performance improvements.
