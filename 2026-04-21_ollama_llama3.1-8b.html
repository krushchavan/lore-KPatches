Last comment 2026-04-22 Re: [PATCH v1 00/17] fuse: extend passthrough to inode operations Needs Work LLM (per-reviewer) Accepted

This patch extends the passthrough functionality in the fuse file system to support inode operations, allowing for improved performance and reduced latency in certain use cases. The main goal is to eliminate round trips for getattr and open operations, which can be a bottleneck for short-lived services and batch jobs that need to fetch large data over the network. The patch aims to add a lazy materialization layer using fuse that supports on-demand fetching of cold data, switching to passthrough mode once the required content has been fetched locally. This will provide near-native performance with minimal userspace overhead. The initial implementation focuses on read-only operations and includes support for open + read passthrough, getattr/statx passthrough, and readdirplus. Future work may include additional features such as handling cross-boundary moves and other edge cases.

Ongoing Submitted 2026-04-16 🔥 TODAY [PATCH v5] btrfs: add BTRFS_IOC_GET_CSUMS ioctl v2 → v3 → v4 → v5 Needs Work LLM (per-reviewer) Under Review

This patch adds a new ioctl (BTRFS_IOC_GET_CSUMS) to the Btrfs file system, allowing userspace to query the on-disk checksums for a specific file without requiring root privileges. The ioctl takes a struct btrfs_ioctl_get_csums_args containing the offset and length of interest, as well as a buffer to store the results. The kernel returns a struct btrfs_ioctl_get_csums_entry indicating whether the extent has checksums available, along with the checksum data if applicable. This feature is intended to speed up mkfs.btrfs operations by avoiding redundant checksum calculations, particularly when using the --reflink option.

Ongoing Submitted 2026-04-16 🔥 TODAY [PATCH] btrfs: fix double-decrement of bytes_may_use in submit_one_async_extent() Needs Work LLM Under Review

The patch addresses a double-decrement issue in the btrfs file system, specifically in the submit_one_async_extent() function. The problem arises when the EXTENT_DO_ACCOUNTING flag is used, causing bytes_may_use to be decremented twice. To fix this, the patch replaces EXTENT_DO_ACCOUNTING with EXTENT_CLEAR_META_RESV, mirroring a similar change made in cow_one_range().

Last comment 2026-04-22 Re: [RFC PATCH v2 2/4] mm/zsmalloc: introduce zs_free_deferred() for async handle freeing Needs Work LLM (per-reviewer) RFC

This patch introduces a new function zs_free_deferred() for asynchronous handle freeing in zsmalloc. The original implementation of locking, grabbing a handle, and unlocking one at a time is inefficient and can be improved by using a workqueue or per-cpu-fying the batches/queues to reduce concurrency issues. An alternative approach suggested is to use two arrays of handles, swapping them under a lock when the defer worker is woken up, and then freeing the handles in the old array.
