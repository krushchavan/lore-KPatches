[PATCH 0/5] allocate extent_buffer GFP_NOFAIL with unlocked retry Neutral llm-chunked-no-replies Awaiting Review

This patch series aims to reduce contention and stalls in the Linux kernel by allowing certain memory allocations, specifically for extent_buffers used by the Btrfs file system, to be retried outside of critical sections when they fail due to lack of memory. The approach taken is to allocate memory with GFP_NOWAIT (which does not block) initially, and if that fails, retry the allocation with GFP_NOFS | GFP_NOFAIL (which will block but guarantees success) without holding any locks. This should help reduce the number of hung tasks and improve system responsiveness.

[PATCH 1/5] btrfs: factor init_extent_buffer from __alloc_extent_buffer Neutral llm-chunked-no-replies Awaiting Review

This patch refactors the initialization of extent buffers in the Btrfs file system. It separates the initialization code from the allocation code, allowing for more flexibility and easier maintenance. The goal is to prepare for preallocating extent buffer data by allocating it together with other necessary resources. This change will make it easier to implement preallocation in the future.

[PATCH 2/5] btrfs: add struct btrfs_eb_prealloc Neutral llm-chunked-no-replies Awaiting Review

This patch adds a new structure called `struct btrfs_eb_prealloc` to carry the extent_buffer and btrfs_folio_state needed for NOFAIL allocations with retries outside the critical section. The existing allocation pathways are refactored to use this new struct, but no functional changes are made. Empty prealloc structs are wired in from callers. This is part of a larger series of patches to support NOFAIL allocations.

[PATCH 3/5] btrfs: enable unlocked NOFAIL retry for eb allocations Neutral llm-chunked-no-replies Awaiting Review

This patch enables unlocked NOFAIL retry for eb allocations in the Btrfs file system. It introduces a new preallocation struct to carry allocation and signal requirements, allowing callers to attempt NOWAIT allocations if supported by the allocator. If the initial allocation fails, it signals that preallocation is required, enabling the caller to perform the necessary allocation and retry with the preallocated resources. The patch also notes that two GFP_NOFS allocations remain under lock and cannot be preallocated due to limitations in the filemap xarray.

[PATCH 4/5] btrfs: probe with GFP_NOWAIT for tree block readahead Neutral llm-chunked-no-replies Awaiting Review

This patch modifies the btrfs readahead mechanism to use GFP_NOWAIT when reading tree blocks, allowing it to fail instead of blocking in direct reclaim if allocation fails. This change is intended to prevent painful stalls and lock contention caused by readahead getting stuck in direct reclaim.

[PATCH 5/5] btrfs: use GFP_NOWAIT when inhibiting eb writeback Neutral llm-chunked-no-replies Awaiting Review

This patch modifies the btrfs_inhibit_eb_writeback function to use GFP_NOWAIT instead of GFP_NOFS when allocating memory for inhibiting writeback on an extent buffer. This change is made because the allocation is already best effort and occurs while holding locks, making NOWAIT a suitable choice.

Needs Work LLM (per-reviewer) RFC

This patch series allows Linux to reserve empty 'standby' NUMA nodes at boot, which can be claimed by drivers at runtime and used for memory tiering or isolation. This is useful for scenarios where hotplugged memory needs to be spread across more nodes than the firmware describes. The patch introduces three ways to reserve standby nodes: a boot parameter, two kernel configuration options, and a mechanism to create these nodes after the topology is finalized. The series includes testing under various topologies and configurations.
