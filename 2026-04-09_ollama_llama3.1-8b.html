[PATCH v4 2/4] btrfs: account for compression in delalloc extent reservation Neutral llm-chunked-no-replies Awaiting Review

This patch addresses an issue in the Btrfs file system where delayed reference reservations were under-estimated due to compression. When compression is enabled, the maximum extent size is reduced from 128MiB to 128KiB, leading to a significant discrepancy between estimated and actual extents. The patch introduces a new function `btrfs_inode_max_extents` to calculate the correct number of extents based on the inode's properties and the maximum extent size. This change ensures that reservations are accurately calculated even when compression is enabled, preventing under-reservation of delayed reference reservations for delalloc writes.

[PATCH v4 3/4] btrfs: make inode->outstanding_extents a u64 Neutral llm-chunked-no-replies Awaiting Review

This patch changes the type of `outstanding_extents` in the Btrfs inode structure from a 32-bit unsigned integer to a 64-bit unsigned integer (`u64`). This is done because the maximum file size, defined by `MAX_LFS_FILESIZE`, can result in extent sizes that exceed the bounds of a 32-bit unsigned integer. The patch also updates related functions and includes additional assertions to ensure correctness.

[PATCH v4 1/4] btrfs: reserve space for delayed_refs in delalloc Neutral llm-chunked-no-replies Awaiting Review

This patch addresses a problem in the Btrfs file system where space is not reserved for delayed references (delayed_refs) during delalloc operations. Delalloc uses a per-inode block reservation to anticipate metadata needs, but it does not account for delayed refs. This can lead to incorrect assessments of available space and poor decision-making in metadata reclaim logic. The patch fixes this by reserving space for delayed refs in the delalloc process, ensuring that these references are properly accounted for during writeback operations.

This patch fixes a bug where the last_unlink_trans update was not being updated when removing a directory in Btrfs. The issue arose from an intentional change to only call btrfs_record_unlink_dir() on success, which could lead to unexpected behavior if btrfs_unlink() does not react correctly to non-zero return values. The patch updates the code to ensure that btrfs_record_unlink_dir() is always called before unlink operations.

🔥 TODAY Re: [RFC PATCH v3 0/4] getting rid of busy-wait in shrink_dcache_parent() Contentious LLM (per-reviewer) Accepted

This patch aims to eliminate busy-waiting in the shrink_dcache_parent function, which is responsible for shrinking dentry lists when processes exit. The current implementation can lead to stalls and crashes due to a use-after-free (UAF) bug. The patch likely addresses this issue by modifying the code path that handles submounts and namespace locking, ensuring that detach_mounts() and namespace_unlock() are not involved in the crash. The problem is related to a complex scenario involving multiple threads, list heads, and dentry traversal, making it difficult to reproduce and debug. The patch is part of an ongoing effort to improve kernel stability and performance.
