Ongoing Submitted 2026-04-09 ⏱ No comments yet [PATCH v4 2/4] btrfs: account for compression in delalloc extent reservation v2 → v3 → v4 Neutral llm-chunked-no-replies Awaiting Review

This patch addresses an issue in the Btrfs file system where delayed reference reservations were under-estimated due to compression. When compression is enabled, the maximum extent size is reduced from 128MiB to 128KiB, leading to a significant discrepancy between estimated and actual extents. The patch introduces a new function `btrfs_inode_max_extents` to calculate the correct number of extents based on the inode's properties and the maximum extent size. This change ensures that reservations are accurately calculated even when compression is enabled, preventing under-reservation of delayed reference reservations for delalloc writes.

Ongoing Submitted 2026-04-09 Last comment 2026-04-10 ⏱ No recent activity [PATCH v4 1/4] btrfs: reserve space for delayed_refs in delalloc v2 → v3 → v4 Positive LLM (per-reviewer) Under Review

This patch addresses a problem in the Btrfs file system where space is not reserved for delayed references (delayed_refs) during delalloc operations. Delalloc uses a per-inode block reservation to anticipate metadata needs, but this calculation does not account for delayed refs. As a result, when writeback occurs and delayed refs are generated, they draw from an under-reserved trans_handle's delayed_refs_rsv, leading to incorrect assessments of reserved space in the metadata space reclaim logic. The patch aims to fix this issue by reserving space for delayed refs during delalloc operations.

The patch aims to preserve write protection across UFFDIO_MOVE by checking the uffd-wp condition of the source PTE before setting writable on the destination. This change is necessary because mremap() preserves the source write state unconditionally, but userfaultfd was not doing so.
