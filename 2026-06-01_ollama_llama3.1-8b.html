This patch limits the value returned by compact_gap() to prevent over-reserving memory for compaction, which can lead to unnecessary reclaim and failed compaction suitability checks. The current implementation returns a gap that scales exponentially with order, but is bounded by COMPACT_CLUSTER_MAX. By capping this value at COMPACT_CLUSTER_MAX, the patch aims to improve compaction success rates and reduce unnecessary reclaim on fragmented production hosts.

Last comment 2026-06-02 [PATCH v2] mm/list_lru: drain before clearing xarray entry on reparent v1 → v2 Needs Work LLM (per-reviewer) Under Review

This patch fixes a concurrency issue in the list_lru code by draining the per-node lists before clearing the xarray entry on reparenting. The problem occurs when a concurrent thread sees the xarray entry as NULL and walks to the parent, causing corruption of adjacent items' links. To fix this, the order is reversed: each per-node list is reparented and marked dead (LONG_MIN) before the xarray entry is cleared. This ensures that any concurrent list_lru operation will either take the dying memcg's per-node lock or see LONG_MIN and walk to the parent, where the items now live.
