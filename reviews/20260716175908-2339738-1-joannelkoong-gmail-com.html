Patch summary
This patch series adds support for io-uring buffer pools to the FUSE file system, allowing servers to pass in a contiguous chunk of memory that the kernel can use to service requests. This decouples the request and payload buffers, reducing memory usage requirements and enabling optimizations for assigning payload memory. Additionally, the series introduces zero-copy functionality, which allows servers with CAP_SYS_ADMIN privileges to directly access client pages or page cache folios without copying data through an intermediary buffer. The zero-copy feature requires using buffer pools and has a dependency on io-uring registered bvec changes. The patch series includes benchmark results showing significant throughput improvements for direct-I/O operations, particularly when backing I/O is not the bottleneck.

Show original comment 
With async INIT this is possible by waiting until the write(2) syscall
for the INIT reply returns.  This is because the initialization is
performed within the context of the write call.

For sync INIT the initialization is performed in the context of the
mount/FSCONFIG_CMD_CREATE, libfuse can wait until the respective
syscall returns.

So yes, it's possible to ensure this from userspace, no kernel change is need.
