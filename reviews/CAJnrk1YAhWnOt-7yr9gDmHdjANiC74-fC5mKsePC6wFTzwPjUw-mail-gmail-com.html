Patch summary
This patch fixes a hang issue in the Linux kernel's fuse module, specifically when using fuse-overlayfs on tmpfs and suspending the system. The problem occurs because the wb_wait_for_completion function is waiting for writeback to complete on an inode that has already been marked as dirty and under writeback. To fix this, the patch adds a check to skip inodes with no data integrity guarantees (e.g., FUSE mappings) when syncing. This approach ensures that the system does not hang indefinitely while waiting for writeback to complete on such inodes.
Joanne Koong (author) 2026-03-17 Needs Work LLM
Author Joanne Koong acknowledged that the issue is not due to the previously proposed fix, but rather the fuse daemon being stuck and unable to service the ->write_inode() call. She provided a stacktrace showing the kernel suspend code happening before freezing user space processes, which contradicts her previous assumption. Author acknowledged that the changes from commit f9a49aa302a0 are still needed, even with the new patch, and agreed to keep them in v2. Author confirms that the original patch's fix is still applicable to the issue, despite the root cause being a cgroup freezer problem.

In the kernel suspend code though, the sync (pm_sleep_fs_sync())
happens before freezing any userspace processes
(suspend_freeze_processes()) so it's odd for the fuse daemon to be
stuck.

It turns out the daemon is stuck because systemd freezes the user
session cgroups first before invoking the kernel suspend:

---

fuse-overlayfs -o ...

---

cgroup: /user.slice/user-1001.slice/session-3.scope
1
frozen 1

Even though the root cause is the cgroup freezer rather than the
folio_prepare_writeback() scenario, the fix proposed previously is
still the fix for this.
