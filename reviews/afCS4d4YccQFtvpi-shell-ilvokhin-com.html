[PATCH 1/8] mm: use zone lock guard in reserve_highatomic_pageblock()

View on lore.kernel.org →
Submitted: 2026-04-28
Active on: 2026-04-28
Patch summary
This patch reworks the cleanup flow for reserve_highatomic_pageblock() to eliminate a branch that depends on compiler inlining decisions, thereby simplifying the code and reducing the likelihood of regressions.
Dmitry Ilvokhin (author) 2026-04-28 Needs Work LLM
The author is addressing Peter Zijlstra's feedback on eliminating the branch in free_pcppages_bulk() by removing __GUARD_IS_ERR(). The author retested the patchset and found that the NULL check is not eliminated, even when using GCC 16. They suspect that function calls between construction and destruction prevent the compiler from proving .lock is non-NULL across the scope. Author addressed Peter Zijlstra's concern about the need for a NULL check in the destructor, explaining that it is unnecessary because the guard infrastructure prevents user code from clearing the lock after construction and before destruction.

So, NULL check is not eliminated and in addition compiler couldn't merge
two almost identical epilogues here. I suspect the function calls

---

My thinking process was the following: the constructor doesn't check for
NULL either, so if something is passing NULL to begin with it should
already crash in constructor not even reaching the destructor.

The only case when we'd need the NULL check in the destructor is if
something changes .lock to NULL after construction, but before
destruction.

This can't happen:

- The guard infrastructure provides no API to clear a guard after
  construction (no_free_ptr()/retain_ptr() only work with __free, not
  guards).

- guard() creates an anonymous variable via __UNIQUE_ID, so user code                                                   
  can't reference it to modify .lock directly.

- CLASS_INIT() is the only macro that allows bypassing the constructor,                                                 
  but it is only used for fd_prepare in include/linux/file.h, not for
  any lock guards.

- Conditional (_try) variants have their own destructor via
  EXTEND_CLASS_COND() that checks __GUARD_IS_ERR() and returns early
  before reaching the base destructor. They are not affected by this
  change.
