Patch summary
This patch series improves large folio readahead and alignment for executable memory by introducing a generic, architecture-independent solution to replace the arch-specific exec_folio_order() function. The patches address two main issues: the mmap_miss heuristic that disables readahead after 100 page faults, and the suboptimal static order returned by exec_folio_order() for non-4K page sizes. The new approach uses preferred_exec_order() to sync mmap_readahead with zone high watermarks, and extends max alignment to mapping_max_folio_size().

Usama Arif (author) 2026-03-27 Neutral LLM
The author addressed a concern about the effectiveness of aligning binaries beyond what the page cache will allocate, explaining that doing so would only reduce ASLR entropy for no benefit on filesystems like btrfs that report PAGE_SIZE.
Show original comment 
mapping_max_folio_size() reflects what the page cache will actually
allocate for a given filesystem, since readahead caps folio allocation
at mapping_max_folio_order() (in page_cache_ra_order()). If btrfs
reports PAGE_SIZE, readahead won't allocate large folios for it, so
there are no large folios to coalesce PTEs for, aligning the binary
beyond that would only reduce ASLR entropy for no benefit.

I don't think we should over-align binaries on filesystems that can't
take advantage of it.
