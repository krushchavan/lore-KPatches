Patch summary
This patch adds a range lookup feature to the zswap compression algorithm, allowing it to handle large folios by scanning the covered range and checking if any slots are still in zswap. If no slot is found, it returns -ENOENT so the caller can read from the backing device. If any slot is present, it fails the large-folio read to prevent returning partially initialized data. The patch also introduces a new function `zswap_is_present` to enable PMD swap-entry consumers to make range decisions before attempting PMD-order swapin.

Yes, so we can end up in a race, which should be handled gracefully.

For example, lets say we have zswap writeback enabled, which means we
can end up in a state where we have a PMD swap entry and 511 of the 512
slots have been written to disk, but 1 slot (slot X) is still in zswap.

We can then have the following race:


  CPU A: PMD swap-in                  CPU B: split-PTE swap-in
  ------------------                 ------------------------

  Checks swap cache: empty

                                       Faults slot X
                                       Adds order-0 folio F
                                       zswap_load(F):
                                         removes X from zswap
                                         marks F dirty

  Checks zswap range: empty
  A is preempted

                                       Unmaps/reclaims F
                                       zswap_store(F):
                                         puts X back in zswap
                                       Removes F from swap cache

  A resumes
  Allocates large swap-cache folio G
  zswap_load(G) finds X in zswap



The correct action would be to reject the PMD order read and fall back
to per-page loading.
