Patch summary
This patch series introduces per-memcg-lruvec zswap accounting by moving memory-cgroup awareness to the zsmalloc layer, allowing for more accurate tracking of compressed memory usage and enabling cgroups to control where their zswapped objects live. The changes also enable future consumers to account memory statistics using a new compressed object memory accounting infrastructure. The patch series includes 11 patches that make various modifications to the zswap and zsmalloc code to achieve this goal, including introducing a new per-zspage array of objcg pointers to track per-memcg-lruvec memory usage. The changes are expected to improve performance by reducing remote access latency costs and enabling more effective reclaim when pressure is localized to a single cgroup.
Joshua Hahn (author) 2026-03-11 Neutral LLM
The author addressed a concern about inconsistent naming conventions in the zsmalloc code by renaming zs_object_copy to zs_obj_copy, and also fixed an issue where find_alloced_obj was returning a signed int instead of an unsigned int.

2.52.0

---

Introduce 3 new fields to struct zs_pool to allow individual zpools to
be "memcg-aware": memcg_aware, compressed_stat, and uncompressed_stat.

memcg_aware is used in later patches to determine whether memory
should be allocated to keep track of per-compresed object objgs.
compressed_stat and uncompressed_stat are enum indices that point into
memcg (node) stats that zsmalloc will account towards.

In reality, these fields help distinguish between the two users of
zsmalloc, zswap and zram. The enum indices compressed_stat and
uncompressed_stat are parametrized to minimize zswap-specific hardcoding
in zsmalloc.

the accounting for how much memory it has compressed. However, when a

each backing zspage with the ratio of memory it backs for the
compressed object.

track the value in bytes and adjust its accounting accordingly.
