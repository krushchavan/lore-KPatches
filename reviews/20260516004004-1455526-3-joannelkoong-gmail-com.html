Patch summary
This patch prepares the fuse filesystem to pass through inode operations by adding a new field 'ops_mask' to the FUSE_DEV_IOC_BACKING_OPEN ioctl input argument. The ops_mask declares which inode operations will be passed through to the backing file once it's attached to the fuse inode on lookup. Currently, only regular files are allowed for read/write/splice/mmap passthrough, but this patch relaxes that restriction by allowing non-regular backing files when setting operations other than FUSE_READ/FUSE_WRITE in ops_mask. Additionally, a filesystem with FUSE_PASSTHROUGH_INO is required to set up passthrough of any inode operations for simplification of inode attribute caching.

Do you think we should couple the passthrough op code to the fuse
opcode so closely, instead of defining the passthrough ops separately,
eg doing something like this?:

#define FUSE_PASSTHROUGH_READ         (1 << 0)
#define FUSE_PASSTHROUGH_WRITE       (1 << 1)

In the (far) future when some more advanced passthrough features get
added (eg full subtree passthrough), it seems like we'd want to add a
passthrough op for that, but that wouldn't map to a fuse op. I think
there are also some fuse ops we might skip defining as passthrough ops
but are implicitly passed through (eg FUSE_STATX which is covered by
FUSE_GETATTR), so it seems more intuitive to define the passthrough
ops as describing capabilities rather than implying that it describes
what specific opcodes get passed through?

Amir Goldstein 2026-05-16 Inline Review Reviewed-by Neutral LLM
Reviewer Amir Goldstein requested that the _OP_ suffix not be dropped, citing the need to distinguish it from other constants and potentially using a larger data type for ops_mask. Reviewer questioned the duplication of information between the flags field and the ops_mask field in the fuse_backing_map structure, suggesting it is unnecessary to include flags in ops_mask. Reviewer Amir Goldstein noted that the ops_mask field in FUSE_DEV_IOC_BACKING_OPEN ioctl could also be used for other operations like FUSE_CREATE and FUSE_CREAT, and suggested leaving it as a u64 to avoid potential issues with reserved space. The reviewer suggested replacing the hardcoded arithmetic in UAPI with a macro-based approach, providing examples of how to use macros for ease of use and suggesting that the current implementation is limiting.
