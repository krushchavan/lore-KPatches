Patch summary
This patch series converts the iova (I/O virtual address) allocation code to use a maple tree data structure, which improves the complexity of the alloc_iova function from O(n) to O(log n). This change is motivated by issues in production workloads where linear searches caused soft lockups. The patches also include self-tests for the iova code and were tested on various systems.

If the in-place marker store fails and WARN_ON_ONCE fires:

  - free_iova_mem(iova) is never reached \u2014 the struct iova leaks.
  - deferred_lo/hi is not updated \u2014 the address range stays permanently
  occupied with no way to reclaim it.
  - After the first occurrence, WARN_ON_ONCE suppresses the warning on all
  subsequent hits, so further failures on this path are completely silent.

  I understand the comment says this cannot happen, and if that guarantee holds
  then it's fine. But if it can, a BUG_ON might be a cleaner way to express
  that invariant \u2014 it makes the intent unambiguous rather than leaving the
  impression the failure is being gracefully handled when it isn't.

  Happy to be wrong if I'm missing something about the maple tree's in-place store
  guarantees.
