Patch summary
This patch is a response to another patch that attempts to add zero-copy functionality over io-uring for the fuse filesystem. However, it points out that the current implementation has a flaw: when an abort occurs in userspace while a zero-copy request is still in-flight, the kernel will skip the necessary logic due to the absence of a valid command associated with the entry.

Baokun Li 2026-06-09 Inline Review Reviewed-by Neutral LLM
Reviewer Baokun Li noted that when a zero-copy request is in-flight in userspace and an abort occurs, the entry's cmd field will be NULL due to fuse_uring_send() completing the command before the entry enters FRRS_USERSPACE, causing the if (cmd) guard to skip unregistering the buffer table slot. Baokun agreed that introducing complexity for an independent handle to the ring ctx is not worth it and accepted leaving the buffer table slot occupied until the ring is destroyed as a reasonable tradeoff.

Hi Joanne,

You're right, I missed that. fuse_uring_send() completes the cmd
and sets ent->cmd = NULL before the entry enters FRRS_USERSPACE.
So when abort hits a zero-copied entry, cmd is already gone \u2014 the
if (cmd) guard skips the unregister entirely.

The timeline is:

 1. Request dispatched \u2192 io_buffer_register_bvec(), zero_copied = true
 2. fuse_uring_send() \u2192 ent->cmd = NULL, io_uring_cmd_done()
 3. Server doing zero-copy IO, ent in FRRS_USERSPACE, no cmd
 4. umount -f \u2192 teardown \u2192 cmd == NULL \u2192 unregister never fires

I now understand the problem you described \u2014 to unregister without
cmd, we'd need an independent handle to the ring ctx, and ensuring
its validity is where the complexity lies. I agree that it's not
worth introducing that complexity for this scenario, and leaving the
buffer table slot occupied until the ring is destroyed is a
reasonable tradeoff.

Thanks for taking the time to walk me through this.
