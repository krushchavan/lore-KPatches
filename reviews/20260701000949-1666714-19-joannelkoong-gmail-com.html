Patch summary
This patch updates the iomap documentation to reflect the new iomap_next model, where every iomap operation takes a single iomap_next callback directly. The callback finishes the previous mapping and produces the next one. Most filesystems build it from begin and end helpers via the iomap_process helper. This change replaces the separate ->iomap_begin() and ->iomap_end() callbacks used in older versions of iomap.
Darrick Wong 2026-07-02 Inline Review Reviewed-by Neutral LLM
Reviewer Darrick Wong noted that the iomap_next model does not allow for the ->begin method to set the private field and the ->end method to dispose of it, as the next ->begin would pass a new mapping to the next ->begin. He suggested making iomap_iter_continue preserve the iomap/srcmap private pointer across memset(0) calls or changing the signature of iomap_process to avoid passing iomap/srcmap explicitly. Reviewer Darrick Wong noted that the documentation still uses '->iomap_begin' and '->iomap_end' callbacks in some places, which should be updated to use 'iomap_next' instead.
