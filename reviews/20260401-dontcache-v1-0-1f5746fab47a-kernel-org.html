Patch summary
This patchset improves write performance when using the RWF_DONTCACHE flag by addressing flush contention issues that occur during heavy write activity. The patches modify the Linux kernel's mm module to alleviate this contention, resulting in better performance for writes. The changes were tested with various workloads and IO modes, including buffered, direct, and unpatched dontcache, and showed significant improvements over the original RWF_DONTCACHE implementation. The patchset also includes benchmarking scripts to demonstrate the results of these tests.

Christoph Hellwig noted that setting nr_to_write in addition to .range_start and .range_end in writeback_iter() can lead to writing every page in the range, which is unnecessary if nr_to_write is set to a value less than the range size. He suggested adding a debug check to prevent this combination.
Show original comment 
But nr_to_write should not change anything.  If .range_start and
.range_end are set in a writeback_iter() loop, writeback_iter will try to
get and writeback every page in the range.  Setting nr_to_write in
addition to that could only reduce the amount written if it was less than
the size of the range, which in your patch it isn't.

In fact we should probably have a debug check to never set both a range
and nr_to_write as that combination doesn't make sense.

Christoph Hellwig 2026-04-05 Inline Review Reviewed-by Needs Work LLM
Christoph Hellwig noted that the patch introduces lock contention between multiple threads accessing shared resources, which can lead to scalability issues, and suggested that this is a problem that has been addressed in other areas of the kernel by separating data and metadata writeback. Christoph Hellwig noted that the writeback thread is not properly handling dontcache writes and suggested queuing up writeback work for each range, possibly by increasing a threshold instead of creating a new wb_writeback_work for each range Reviewer noted that the 30s timeout for IOCB_DONTCACHE writeback is actively harmful, especially on SSDs and HDDs, and suggested reconsidering the current timeout value.
Show original comment 
Well, an any time you hit a shared resources from multiple threads you
create that lock contention.   Which is why in file system and writeback
land we've moved away from random user processes hitting both data and
metadata (e.g. XFS AIL) writeback as it leads to these scalability
issues.  At some point we might run out of steam in a single thread,
although so far that's mostly been because it does stupid things
(e.g. writeback on file systems doing complex allocator stuff).

---

And that is called the writeback thread.  So what we should do there
is to make sure we queue up writeback on it for each dontcache write.
Initially queuing up a wb_writeback_work for each range might be first
approximation, although we should probably find a way to just increase
a threshold if going down that road.

---

Yes, we've had lots of observation that the current 30s timeout is
actively harmful.  Especially on SSDs, but even on HDD just keeping
the active might make sense.
