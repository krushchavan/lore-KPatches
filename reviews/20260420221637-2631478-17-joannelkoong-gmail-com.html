Patch summary
This patch adds a new feature to the FUSE (Filesystem in Userspace) kernel module called 'passthrough open'. It allows the server to indicate that certain operations should be skipped and instead performed directly on the backing file. Specifically, when the FUSE_PASSTHROUGH_OP_OPEN flag is set, the kernel will skip sending FUSE_OPEN/FUSE_OPENDIR requests to the server and open the backing file directly. This also implies passthrough readdir for directories and read/write operations for regular files. Additionally, if the server handles truncation logic in its FUSE_OPEN handler, the vfs layer will call handle_truncate() which may trigger a fuse_setattr() call to truncate the backing file directly or send a FUSE_SETATTR request to the server.

FWIW, those are all optional operations that already have
a default kernel implementation, so I suggest that OPEN passthrough
implies that we just skip them, same as you did for FLUSH.

It is worth mentioning that the atomic open code path still goes
through the server, which means that server could return
FUSE_PASSTHROUGH_OP_OPEN entry and either opt-in
for FOPEN_PASSTHROUGH or not and also that the server does
provide ff->fh and does expect release in this case.

Semantically this seems fine, the only thing to watch out for is
that because FUSE_PASSTHROUGH_OP_OPEN is for the lifetime
of the inode, it is almost as if ff->passthrough_open is redundant,
only it is not, for this first atomic open.

Maybe the mental model would be simpler if atomic open
would send RELEASE to the server and use the
FUSE_PASSTHROUGH_OP_OPEN semantics
(force FOPEN_PASSTHROUGH and no lseek/ioctl/flock) for this
first open as well, to be more consistent.

Thanks,
Amir.

Amir Goldstein 2026-04-20 Inline Review Reviewed-by Neutral LLM
Reviewer Amir Goldstein noted that if passthrough open is enabled, the server should not handle truncate and write operations, suggesting making conflicting fuse configurations mutually exclusive. Reviewer Amir Goldstein argued that passthrough open should be considered an inode operation, similar to CREATE, because it operates on the backing inode, but acknowledged that this perspective is confusing and suggested keeping the current implementation. Reviewer questioned whether the backing inode needs to be opened with O_TRUNC when FUSE_PASSTHROUGH_OP_OPEN is set, pointing out that if it is, the file will be truncated and fuse will read the size attribute. Reviewer noted that if the server doesn't see the open and doesn't provide ff->fh, then file operations carrying fh cannot be sent to the server, suggesting additional work is needed for passthrough open Reviewer noted that the patch does not address all previously raised issues and suggested further discussion on how to resolve them without overly complicating the code.
