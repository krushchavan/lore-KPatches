Patch summary
This patch reserves space for delayed references in the delalloc mechanism of Btrfs, addressing a potential issue with inode->block_rsv tracking. The author considers alternative approaches to optimize struct btrfs_inode size by allocating reserved space indirectly or using existing fields.

Filipe Manana 2026-03-25 Reviewed-by Needs Work LLM
Reviewer noted that re-computing delayed_refs is not expensive and can be done in a workqueue, but increasing the structure size to track it would be undesirable if it stays under or at 1024 bytes. Reviewer noted that adding an allocation for inode->delayed_rsv could lead to -ENOMEM failures and questioned the benefits of reducing struct btrfs_inode size by using larger page sizes. Reviewer noted that embedding inode->block_rsv directly in struct btrfs_inode provides a memory allocation savings and potentially better cache line spatial locality, but only benefits when the page size is greater than 4K.

Re-computing (calling btrfs_calc_delayed_ref_bytes()) is not
expensive, and it's done in workqueue for finishing ordered extents.
I wouldn't worry about that. But tracking it somewhere, implies
increasing the structure size - if it stays under or at 1024 bytes,
it's fine.

---

Adding an allocation adds potential for -ENOMEM failures.
And for 4K pages we will still be able to allocate 4 inodes per page,
so no gains here.
For a 64K page we can get a few more inodes (7 or 8).

---

The reason we have it embedded it's because it saves a memory
allocation, and it may also be faster and maybe better cache line
spatial locality.
In the end we won't get any benefit unless the page size is > 4K so
that we can get more inodes per page.
