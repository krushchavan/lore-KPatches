[PATCH v4 0/7]] VFS: Prepare to lift lookup out of exclusive lock for directory ops

View on lore.kernel.org →

Patch summary
This patch addresses a potential lock imbalance issue in the VFS directory operations by ensuring that the parent lock is reacquired before exiting early from certain lookup functions. The problem arises when d_alloc_parallel() or d_alloc() return without holding the lock, causing the caller to unlock an unheld rwsem and corrupting its state. To fix this, the patch modifies the early return paths in these functions to reacquire the parent lock before returning.
NeilBrown 2026-04-30 Inline Review Reviewed-by Needs Work LLM
NeilBrown noted that the parent lock is dropped unconditionally before d_alloc_parallel() and three early return paths exit without reacquiring it, leading to a potential corruption of the rwsem state when inode_unlock_shared() is called after ->lookup() returns. He suggested moving the unlock/relock around d_alloc_parallel(), which resolves the issue.

From: NeilBrown <neil@brown.name>
Subject: [PATCH] VFS/xfs/ntfs: drop parent lock across d_alloc_parallel() in
 d_add_ci()

A proposed change will invert the lock ordering between
d_alloc_parallel() and inode_lock() on the parent.
When that happens it will not be safe to call d_alloc_parallel() while
holding the parent lock - even shared.

We don't need to keep the parent lock held when d_add_ci() is run - the
VFS doesn't need it as dentry is exclusively held due to
DCACHE_PAR_LOOKUP and the filesystem has finished its work.

So drop and reclaim the lock (shared or exclusive as determined by
LOOKUP_SHARED) to avoid future deadlock.
