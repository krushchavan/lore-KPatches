Patch summary
This patch proposes containerizing ext4 for safer operation, allowing for isolation and security benefits without requiring near-native performance. The approach involves using lklfuse, which is a user-space filesystem that uses the Linux kernel's virtual machine (VM) to run a kernel instance in user space. This method has pros such as easier setup and maintenance compared to implementing a fuse-iomap server for every filesystem, but may have worse performance. If near-native performance is not a requirement, using lklfuse seems like a less painful option.
Darrick Wong 2026-03-16 Inline Review Reviewed-by Needs Work LLM
Reviewer Darrick Wong noted that lklfuse uses user-mode-linux (UML) based on arch/lkl/Kconfig, and suggested an alternative approach by modifying libguestfs to run the fuse server inside a systemd container instead of directly from the CLI. reviewer clarified that near-native performance is not a strict requirement for the patch's use cases, and suggested that lklfuse might be a more maintainable option if performance is not critical reviewer noted that a good non-iomap fuse server for every filesystem would be beneficial, and expressed concern about the robustness of libext2fs compared to the kernel Reviewer noted that using lklfuse would require running a full IO stack, potentially increasing complexity and overhead. Reviewer noted that btrfs copy-on-write can be handled with fuse-iomap, but other features such as fsverity, fscrypt, and data checksumming are not currently supported by iomap. Reviewer Darrick Wong expressed skepticism about the potential benefits of using lklfuse, suggesting that it may be easier to maintain than fuse-iomap but also implying that its performance might not be significantly worse.

---

Anyone with strong verified boot requirements is going to want this.
The threat model used by Android and Chromium OS is that all on-disk
state is *never* trusted.  The root filesystem is on a dm-verity
protected partition, so mounting it in the kernel is fine.  However,
the writable partition should be mounted in userspace.

I checked with the ChromeOS security team, and they do consider being
able to write corrupted metadata to disk that compromises the system
on the next reboot to be a vulnerability.

This is also a case where performance actually matters, including
for F2FS on zoned UFS which is always copy-on-write for user data.
