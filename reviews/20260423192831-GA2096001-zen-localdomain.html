Patch summary
This patch addresses an issue where deleted snapshots remain in the quota system due to a problem with assigning ownership of data blocks. The proposal is to add an ioctl that allows userspace to assign extents or files to a specific subvolume, rather than having the kernel make arbitrary decisions. This approach aims to avoid lossy processes and potential inconsistencies in metadata ownership.

Reviewer noted that in cases where a block is owned by multiple subvols, it's acceptable to assign the block to the subvol with the lower number, and suggested this as a simple solution for cleaning up unowned blocks. Reviewer suggested a simple solution to resolve ownership of data blocks: picking the lowest ID subvolume as owner, which would be sufficient for this patch despite not being perfect The reviewer, Marc MERLIN, expressed frustration with the current state of squotas in Btrfs, where deleted snapshots remain and cause clutter in the quota display. He acknowledged that perfect accounting is not possible due to shared blocks, but wants a simple solution to assign ownership of unowned extents to the remaining subvolumes.
