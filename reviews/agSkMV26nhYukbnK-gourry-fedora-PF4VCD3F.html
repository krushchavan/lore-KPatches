Patch summary
This patch series introduces a new flag __GFP_UNMAPPED to the Linux kernel's memory allocation system, allowing for more efficient handling of unmapped memory allocations. The current approach involves adding an additional level of indirection, increasing the migratetype matrix. An alternative solution proposed by Gregory Price is to have the buddy unmap memory after zeroing and allocation, and remap it on free. This would simplify the process and hide the changes from existing users.

Brendan Jackman 2026-05-13 Inline Review Reviewed-by Needs Work LLM
Reviewer noted that the current design only supports one migratetype being unmapped, and requested a more general solution that can handle arbitrary migratetypes. Reviewer noted that the proposed __GFP_UNMAPPED mechanism may not be universally applicable due to the presence of guest memory, where a user address is not available for write() operations. Reviewer noted that the proposed __GFP_UNMAPPED mechanism is not primarily about default memory state, but rather about batching and reducing TLB flushes during allocation; they suggested unmap-on-alloc approach as a cleaner alternative to achieve this goal. Reviewer noted that doing unmapping on a per-allocation basis would result in most map-unmap cycles being pointless, as most pages are expected to be unmapped, and suggested considering the opposite mechanism of unmap-on-free instead.

That would be cleaner indeed, but the key question here isn't about the
default state of memory here, it's about batching.

The reason we need to do it at the block granularity is that a TLB flush
every time we allocate one of these pages is a performance nonstarter -
that's actually the entire point of this series. If you can afford a TLB
flush per allocation then you don't need __GFP_UNMAPPED for the
guest_memfd usecase, the existing direct map removal series [0] is
already fine.
