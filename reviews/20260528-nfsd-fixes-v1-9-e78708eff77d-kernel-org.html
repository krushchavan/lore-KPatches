[PATCH 09/10] nfsd: cap decoded POSIX ACL count to bound sort cost

View on lore.kernel.org →
Submitted: 2026-05-29
Active on: 2026-05-29
Patch summary
This patch prevents a potential denial of service attack on the NFS server by capping the count of decoded POSIX ACL entries at a fixed maximum, rather than allowing it to be chosen by the client. The problem arises from an O(n^2) bubble sort used in the server's compound processing path, which can lead to unbounded CPU usage if the client chooses an excessively large count. By rejecting counts greater than NFS_ACL_MAX_ENTRIES before any allocation or sorting occurs, the patch bounds the number of comparisons and prevents the potential denial of service attack.
Chuck Lever 2026-05-29 Reviewed-by Neutral LLM
Reviewer Chuck Lever pointed out that NFS4ERR_RESOURCE is not a valid error code for this scenario, as it only exists in NFSv4.0; instead, the correct error codes to use are nfserr_rep_too_big or nfserr_rep_too_big_to_cache for NFSv4.1+, and suggested using NFS4ERR_INVAL to indicate invalid input.
Show original comment 
Among other reasons, NFS4ERR_RESOURCE does not exist outside of NFSv4.0.
The server is not out of resources in this case.

For NFSv4.1+ NFSD's compound encoder converts it to either
nfserr_rep_too_big or nfserr_rep_too_big_to_cache, neither of which are
close to appropriate.

Rick's NFSv4 POSIX ACL specification uses NFS4ERR_INVAL for just about
every error condition. The specific meaning of INVAL is "invalid input"
which is exactly what we have here.

nfsd4_encode_posixacl(struct xdr_stream *xdr, struct svc_rqst *rqstp,

                return nfserr_resource;
[...]

...so if you set an ACL that is longer than NFS_ACL_MAX_ENTRIES you
already can't retrieve it with NFSv4. Given that, I went with the above
