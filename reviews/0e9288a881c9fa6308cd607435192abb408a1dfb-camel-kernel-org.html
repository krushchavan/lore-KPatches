Patch summary
This patch addresses an issue where the nfsd_nl_cache_flush_doit function hangs, potentially causing a lockup warning. The problem is likely due to a large number of listeners, which can cause the registration process to take too long. To solve this, the patch proposes implementing a more structural solution, possibly involving asynchronous processing of registration requests or keeping a queue of requests and processing them in order from a dedicated thread.
NeilBrown 2026-07-22 Inline Review Reviewed-by Needs Work LLM
NeilBrown questioned the need for a mutex across multiple netlink registrations, suggesting that it could cause issues with out-of-order requests and proposing an alternative solution using a queue of requests processed in order from a dedicated thread or by driving async RPC submissions serially. NeilBrown noted that svc_register() prints a warning and gives up on other versions/programs when registration fails, causing the listener not to get added if no listeners got registered, which prevents nfsd from starting; he suggested registering in a separate thread and retrying on failure following a normal rpc fail/retry pattern Reviewer suggested using a single-threaded work-queue per net-namespace to handle listener registration and de-registration, noting that this approach is already used for sync-canceling the work-item before reusing it.

svc_register() prints a warning, then gives up on other
versions/programs.
if svc_setup_socket() get an error from svc_register() it aborts, which
ripples back to svc_addsock() aborting, so the listener doesn't get added.

Then if no listeners got registered, nfsd refuses to start.

It would likely make sense for the registration to happen in a separate
thread and to retry on failure following a normal rpc fail/retry
pattern.

Then if the admin restarts rpcbind, it will start working without
further intervention.

---

A single-threaded work-queue (per net-namespace) seems like a good
solution.  Each listener has a work_item that is used for registration
and de-registration.  We sync-cancel it before re-using it.
