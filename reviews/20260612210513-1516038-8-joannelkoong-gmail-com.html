[PATCH v4 07/10] fuse: add io-uring buffer pools (implemented as buffer ring)

View on lore.kernel.org →
Submitted: 2026-06-16  ·  Last comment: 2026-06-18
Active on: 2026-06-18 • 2026-06-17 • 2026-06-16
Patch summary
This patch adds support for buffer pools to the fuse io-uring implementation, allowing the server to pass in a contiguous allocation of buffers that can be used as needed by the kernel. This decouples the number of entries from the number of buffers required, reducing memory usage and enabling future optimizations such as incremental buffer consumption. The server must set the FUSE_URING_BUFPOOL flag during registration and provide the payload buffer pool through the iovec payload in the first sqe. A buffer ring data structure is used to manage the buffer pool, selecting and recycling buffers on demand.
Joanne Koong (author) 2026-06-16 Neutral LLM
The author is addressing Miklos' concerns about the interface for handling multiple buffer pools, specifically how the server can compute the address of a buffer given its ID and pool information. The author proposes adding a 'uint16_t pool_idx' to struct fuse_uring_ent_in_out to indicate which pool a buffer belongs to, but notes that this would require userspace to keep track of the starting address of each pool.

Looking at Bernd's reduced queue series [1], I think we'd most likely

registration. I think along the lines of the config sqe Bernd brought
up, it seems cleanest to add a new cmd op
(FUSE_IO_URING_CMD_QUEUE_SETUP) where queue-level settings (eg flags,
number of queues, bufpool, etc) gets passed in. This would have the
nice side effect of doing the queue creation in this op instead of
right now creating the queue lazily when the first sqe gets
registered. On the userspace side, this adds a bit more friction (eg a
two-phase setup where userspace submits the queue setup cmd, reaps its
completion, and reuses that sqe to register an entry), but that seems
like a reasonable tradeoff.

Does this sound ok to you and Bernd?

> On Fri, 12 Jun 2026 at 23:07, Joanne Koong <joannelkoong@gmail.com> wrote:

> > Buffer ring request flow
> > ------------------------
> > |  Kernel                                  |  FUSE daemon
> > |                                          |
> > |  [client request arrives]                |
> > |  >fuse_uring_send()                      |
> > |    [select payload buf from pool]        |
> > |    >fuse_uring_select_buffer()           |
> > |    [copy headers to ent]                 |
> > |    >copy_header_to_ring()                |
> > |    [copy payload to selected buf]        |
> > |    >fuse_uring_copy_to_ring()            |
> > |    [set buf_id in ent_in_out header]     |
> > |    >io_uring_cmd_done()                  |
> > |                                          |  [CQE received]
> > |                                          |  [read headers]
> > |                                          |  [read payload from buf_id]
>
> How does buf_id map to a buffer address?

The server knows the starting address of the bufpool (since it
allocated it) and it knows the size of each buffer in the pool. It can
compute from the buf_id the address corresponding to the buffer.

>
> Wouldn't it be simpler to provide the buffer address directly?

This would have to be returned as part of the struct
fuse_uring_ent_in_out. There's currently 12 bytes that are free to
use. Returning buf_id takes up 2 bytes and I think is better than
using 8 bytes to provide the buffer address. However, returning the
buf id makes adding incremental buffer consumption in the future a bit
trickier (eg the server would need to keep track of the offset into
the previously used buffer and do any alignment rounding up). Maybe a
good compromise is returning a uint32_t offset that corresponds to the
offset in the bufpool? The buf size is a uint32_t, so I think this
would work just as well as providing the buffer address directly.

>
> We also need some uAPI documentation (generally not just for this
> patchset).  What about a new directory:
> Documentation/filesystems/fuse/uapi?

That sounds good. I'll write some uAPI documentation for this and add
it to that new directory.

> But which buffer?   The interface is supposed to handle multiple
> pools, in which case the server needs to find the pool and the buffer
> within the pool.

If we later wanted to add support for multiple buffer pools, we would
need to extend the struct fuse_uring_ent_in_out to include a "uint16_t
pool_idx" to indicate which pool. Userspace would be responsible for
keeping track of the starting address of each pool.

>
> FIXED ops also require buf_index.  Things are getting complicated...

For FIXED ops not through zero-copy, there's one index for the entire
pool (eg this is the index where the pool resides). It passes this by
setting the sqe->buf_index.
For FIXED ops through zero-copy, userspace is required to register a
sparse buffer where the kernel should place the application's/page
cache's pages. Userspace determines the index into this sparse buffer
for the sqe and it passes this index to the kernel in the init setup
in the "uint16_t zero_copy_index;" field when it registers the sqe.

Neither require the kernel to pass back a buf index corresponding to
the fixed buffer index. For both cases, userspace knows the index for
the fixed buffer (it is the one who determined it) and it is the one
who passes this info to the kernel at setup time.

>
> > This would have to be returned as part of the struct
> > fuse_uring_ent_in_out. There's currently 12 bytes that are free to
> > use.
>
> Does the 12 bytes limit come from the cqe?

The 12 byte limit comes from the "struct fuse_uring_ent_in_out"
definition in the uapi [1].

>
> > Returning buf_id takes up 2 bytes and I think is better than

For multiple buffer pools, I was thinking that would be through a

> So should that be the "pool index" that would be the same as
> "buf_index" for registered buffers?

I think we should just pass back "pool index" and have userspace keep
track of the registered buffer id that corresponds to that pool index.

>
> On Wed, 17 Jun 2026 at 14:50, Joanne Koong <joannelkoong@gmail.com> wrote:
>
> > I'm not really convinced there's a compelling use case for needing
> > multiple bufpools in one queue. This was the discussion with Bernd in
> > [1] about this. In my opinion, incremental buffer consumption and
> > allowing different queues to support different payload sizes would
> > address the use cases of multiple bufpools within one queue, but would
> > be simpler.
>
> Confused.  I thought this was exactly about multiple bufpools
> (containing different size bufs) per ring.
>
> If not, then what's the point of dynamically assigning bufs to sqe's?

imo the main benefit is for incremental buffer consumption (allowing
non-overlapping parts of a buffer to be used by multiple entries
concurrently). To recycle back a buffer that is used by multiple
entries, it would require all requests to have been completed but I
don't think in practice this is a bottleneck with reasonably sized
buffer pools across queues. imo incremental buffer consumption scales
more dynamically with runtime workloads than preallocating fixed
buffers of different sizes across different queues trying to
anticipate / guess workload needs. Overprovisioning small buffers
wastes memory if those requests never appear, while splitting the
memory budget towards smaller pools reduces capacity for concurrent
large requests, and workload patterns might need either at different
times.

I think we should still make the design for bufpools
forward-compatible with supporting multiple bufpools if a use case
comes up in the future, but I think for the majority of
servers/workloads, incremental buffer consumption would meet their
needs and be simpler logic wise and avoids the server having to try to
figure out how to optimally distribute different size bufpools across
differnet queues.

>From my perspective, the use case I see for queues supporting
different payload sizes is for separating IO requests from non-IO
requests (eg lookups, getattrs) onto different queues so that non-IO
requests don't get stuck waiting behind larger IO requests. The non-IO
queues would only need small buffers (eg <= 4k). Userspace could try
implementing separation with threadpools instead of fuse uring doing
queue-level separation, but I think this adds extra context-switching
overhead and is a lot more complex to get performant. If a server
truly needs multiple buffer sizes beyond just non-IO vs IO separation,
I think having multiple bufpools per queue would be a better choice
than dedicating queues to specific categories of buffer sizes; in the
initial discussion, I had assumed the buffer search overhead /
complexity would be worse with per-queue multiple bufpools, but with
the free lists you mentioned, I think you could do the same thing here
with a free list at the fc level for bufpools across all queues, and
that would address the search concern.

Thanks,
Joanne

could just look at the request type, and dispatch accordingly so that
the metadata ops go to the non-io queues and the read/write ops go to
the io queues. I think the dispatch logic would be otherwise the same

> start with an single max-IO buf and a single queue (per numa) and then

I'm not seeing why this would be an issue. For multiple queues of the

different buffer sizes, we'd jjust have two pools of queues (non-io
and io) and select based on request type first which pool then apply
normal queue selection within that pool. I think if a server has needs
for multiple categories of buffer sizes beyond just non-io vs io,
having multiple bufpools per queue makes more sense than dedicated
queues to the specific categories of the different buffer sizes, but I
doubt this level of granularity would be needed.

>

Syscall aggregation still applies within each ring, but I don't think
sysclal aggregation is really a big factor given that libfuse and the
server handles requests as they arrive and respond as soon as the
request is handled, instead of batching multiple responses before
submitting; on the dispatch side, requests are sent one by one as they
come in rather than batched, so there's no aggregation to lose.

the fixed buffer index. For both cases, userspace knows the index for
the fixed buffer (it is the one who determined it) and it is the one
who passes this info to the kernel at setup time.

definition in the uapi [1].
