Patch summary
This patch set improves write performance when using the RWF_DONTCACHE flag by kicking the flusher thread for the superblock instead of initiating writeback directly. This change is based on benchmarking results that show a clear improvement over the status quo, particularly in cases with heavy write activity. The patch also includes a comparison of different IO modes for nfsd and highlights the poor performance of RWF_DONTCACHE with write-heavy workloads. The approach taken involves modifying the generic_write_sync function to kick the flusher thread, which is expected to better isolate the DONTCACHE writeback behavior and improve overall performance.

  This is the biggest regression. With whole-BDI, the flusher did one batched pass through all dirty inodes via writeback_sb_inodes(). With per-inode, each of 4 writers queues a separate work item processed serially by write_inode_now() — losing the batch I/O merging benefit.

  Mixed-mode (Scenario D) is the intended production case and it's essentially identical or slightly better — per-inode writeback creates less device contention for buffered readers.

  Summary

  The per-inode approach is neutral-to-slightly-better for the production scenario (Scenario D), but regresses on multi-writer and random write workloads. The core issue is loss of I/O batching
 — writeback_sb_inodes() processes all dirty inodes in one blk_plug'd pass, while per-inode write_inode_now() calls are processed one at a time. The read regressions likely reflect different
  system conditions since buffered/direct reads also dropped ~20%.

Christoph Hellwig 2026-04-08 Inline Review Reviewed-by Needs Work LLM
Christoph Hellwig noted that the current implementation of wakeup_flusher_threads_bdi causes WB_SYNC_NONE writeback to occur every time a DONTCACHE write finishes, which is inefficient and unnecessary. He suggested introducing a new WB_start_dontcache bit, a get_nr_dontcache_pages() helper function, and other related changes.
Show original comment 
wakeup_flusher_threads_bdi ends up calling wb_start_writeback eventually,
which sets WB_start_all, pushes the reason to start_all_reason and then
does the actual wakeup.

The flusher thread then through wb_check_start_all does a WB_SYNC_NONE
writeback based on get_nr_dirty_pages.  Which seems wrong - we don't
want to do a huge writeback evertime the some DONTCACHE write finished.

So I think you'll want a new WB_start_dontcache bit, a new
get_nr_dontcache_pages() helper on a new node counter, etc.
