Patch summary
This patch adds a new function called untagged_addr_remote_unlocked() to the RISC-V MM module in the Linux kernel. The function is designed to handle untagged addresses for remote processes, which are addresses that do not have the high bits set. This is necessary because some system calls and interfaces, such as /proc/<pid>/cmdline and ptrace peek/poke, may produce tagged addresses, but the process may be in a state where tagging is disabled. The function aims to handle these cases correctly by checking if the address is already untagged and maps to a valid VMA (memory region), and if so, allowing access to the memory without acquiring any locks.

The target VMA might be at 0x12345678, but applying PMLEN 7
to that tagged pointer does not produce that address.

Previously, the order was:

Take mmap read lock.
Read pmlen.
Untag the address.
Look up the VMA.

Changing PMLEN takes the mmap write lock. The read and write
operations were therefore serialized.

The new order is:

Read pmlen without mmap lock.
Untag the address.
Attempt the per-VMA lookup.
Possibly take mmap lock later.
Continue using the already-untagged address.

A concurrent PMLEN change can occur between those operations?
