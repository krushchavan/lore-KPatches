Patch summary
This patch series refactors and extends the io_uring registered buffers infrastructure to allow external subsystems to register pre-existing bvec arrays directly and obtain a pointer to the registered memory region, which is necessary for making fuse zero-copy possible.

Jens Axboe 2026-03-25 Inline Review Reviewed-by Needs Work LLM
Reviewer Jens Axboe noted that the submit locking around io_ring_submit_unlock() may create a false sense of protection, as it does not prevent the registered memory region from becoming invalid immediately after unlocking. He suggested removing this locking and instead returning a struct containing both the pointer to the registered memory region and the number of pages.

This looks suspicious, but I actually think it looks suspicious because

Show original comment 
My understanding is that once a memory region is registered to the
ring, it's registered for the ring's lifetime. There's no uapi to
unregister a memory region and my interpretation of the last paragraph
in this thread [1] is that unregistration is not intended to be
added/supported. I think the submit locking is needed in case another
thread is currently registering it so we don't see partially
initialized state between ptr and nr_pages (eg if the caller calls
this from a task work callback).

[1] https://lore.kernel.org/linux-fsdevel/000f7db7-5546-4680-bef2-84ce740ad8fd@gmail.com/
