Patch summary
This patch adds a deferred free mechanism to handle GFP_ATOMIC failures during IOVA (I/O Virtual Address) allocation in the Linux kernel's iommu subsystem. When mas_store_gfp(NULL, GFP_ATOMIC) fails due to memory pressure, the IOVA entry remains in the maple tree and is added to a lockless per-domain deferred free list. A delayed workqueue retries the erase with GFP_ATOMIC after a 10ms delay, allowing the allocation to succeed when transient memory pressure subsides. This mechanism provides a safety net for rare GFP_ATOMIC erase failures, ensuring that IOVA address reuse and corruption are prevented.
Jason Gunthorpe 2026-06-15 Reviewed-by Neutral LLM
Reviewer Jason Gunthorpe noted that many callers to mas_store_gfp(NULL, GFP_ATOMIC) cannot handle an -ENOMEM return value and requested consideration for this limitation. The reviewer suggested storing NULL in the maple tree instead of failing GFP_ATOMIC allocation, and then dropping a ZERO_ENTRY and setting a bit to trigger a tree sweep on the next allocation if ENOMEM occurs. The reviewer notes that deferring rebalancing in the maple tree leaves available space unavailable and suggests handling this internally or documenting a reasonable pattern for failable erases.
