[PATCH v4 00/10] fuse: add io-uring buffer pools and zero-copy

View on lore.kernel.org →
Submitted: 2026-06-15  ·  Last comment: 2026-06-18
Active on: 2026-06-22 • 2026-06-19 • 2026-06-18 • 2026-06-17 • 2026-06-16 • 2026-06-15
Patch summary
This patch series adds buffer pools and zero-copy to the fuse io-uring implementation, allowing servers to pass in a managed buffer pool that reduces memory usage requirements and enables future optimizations. The decoupling of ents and buffers also eliminates the need for dedicated payload buffers, reducing memory waste. Additionally, the series introduces zero-copy functionality, which allows direct access to client pages or page cache folios without copying data through an intermediary buffer, requiring CAP_SYS_ADMIN privileges and registered buffers.
Joanne Koong (author) 2026-06-15 Neutral LLM
The author addressed Miklos Szeredi's feedback about the buffer pool interface being inconsistent and suggested a new command op (FUSE_IO_URING_CMD_QUEUE_SETUP) to pass in queue-level settings, including the number of reduced queues and bufpool. The author also explained that returning the buffer address directly would require 8 bytes, but using buf_id takes up only 2 bytes and allows for easier incremental buffer consumption in the future.

>
> On Fri, 12 Jun 2026 at 23:07, Joanne Koong <joannelkoong@gmail.com> wrote:
> >
> > Allow servers to use an io-uring registered buffer as the buffer pool.
> > When the server registers buffers with io-uring, the pages backing the
> > buffers are pinned upfront. This eliminates the overhead of
> > pinning/unpinning user pages and translating virtual addresses per i/o
> > request. This also allows servers to use the same registered buffers for
> > subsequent backing store I/O (eg read_fixed/write_fixed), keeping data
> > in the same pinned pages without additional pinning or mapping overhead
> > required.
>
> Are pinned pages accounted and limited?

Yes, pinned pages are accounted and limited by the kernel io-uring
infrastructure (in io_account_mem() in io_uring/rsrc.c) when userspace
registers the buffers in io-uring.

>
> > Registered buffers require the buffer pool (FUSE_URING_BUFPOOL) to be
> > enabled.
>
> Is there any particular reason for this?  Not saying this is bad, just
> wondering...

There's no technical reason it can't but it keeps things simpler (with
the buffer pool, userspace can more optimally register the bufferpool
into one fixed slot since the buffers are contiguous), reduces the
surface area of configurations that need to be tested, and allows the
feature to be advertised under the FUSE_URING_BUFPOOL capability flag
during init instead of needing a separate flag. I think in practice,
servers that would want the benefits from registered buffers would
also want the benefits from buffer pools, so I don't think we're
leaving out a wanted use case.

Thanks,
Joanne

> > To use this, the server must set the FUSE_URING_BUFPOOL flag during
> > registration and provide the payload buffer pool through the iovec
> > payload in the first sqe.
>
> I find this interface inconsistent.   Decoupling the buffer(s) from
> the seq's means that the registration should be decoupled as well, no?

Looking at Bernd's reduced queue series [1], I think we'd most likely
want to also pass in the number of reduced queues separate from sqe
registration. I think along the lines of the config sqe Bernd brought
up, it seems cleanest to add a new cmd op
(FUSE_IO_URING_CMD_QUEUE_SETUP) where queue-level settings (eg flags,
number of queues, bufpool, etc) gets passed in. This would have the
nice side effect of doing the queue creation in this op instead of
right now creating the queue lazily when the first sqe gets
registered. On the userspace side, this adds a bit more friction (eg a
two-phase setup where userspace submits the queue setup cmd, reaps its
completion, and reuses that sqe to register an entry), but that seems
like a reasonable tradeoff.

Does this sound ok to you and Bernd?

Thanks,
Joanne

> > Buffer ring request flow
> > ------------------------
> > |  Kernel                                  |  FUSE daemon
> > |                                          |
> > |  [client request arrives]                |
> > |  >fuse_uring_send()                      |
> > |    [select payload buf from pool]        |
> > |    >fuse_uring_select_buffer()           |
> > |    [copy headers to ent]                 |
> > |    >copy_header_to_ring()                |
> > |    [copy payload to selected buf]        |
> > |    >fuse_uring_copy_to_ring()            |
> > |    [set buf_id in ent_in_out header]     |
> > |    >io_uring_cmd_done()                  |
> > |                                          |  [CQE received]
> > |                                          |  [read headers]
> > |                                          |  [read payload from buf_id]
>
> How does buf_id map to a buffer address?

The server knows the starting address of the bufpool (since it
allocated it) and it knows the size of each buffer in the pool. It can
compute from the buf_id the address corresponding to the buffer.

>
> Wouldn't it be simpler to provide the buffer address directly?

This would have to be returned as part of the struct
fuse_uring_ent_in_out. There's currently 12 bytes that are free to
use. Returning buf_id takes up 2 bytes and I think is better than
using 8 bytes to provide the buffer address. However, returning the
buf id makes adding incremental buffer consumption in the future a bit
trickier (eg the server would need to keep track of the offset into
the previously used buffer and do any alignment rounding up). Maybe a
good compromise is returning a uint32_t offset that corresponds to the
offset in the bufpool? The buf size is a uint32_t, so I think this
would work just as well as providing the buffer address directly.

>
> We also need some uAPI documentation (generally not just for this
> patchset).  What about a new directory:
> Documentation/filesystems/fuse/uapi?

That sounds good. I'll write some uAPI documentation for this and add
it to that new directory.

Thanks,
Joanne

> > |                                           | [issue io_uring READ at
> > |                                           |  ent->zero_copy_index]
>
> FIXED_READ takes an address, right?  How does all this work at the
> userspace level?

The userspace FIXED_READ call (eg set up usually through the liburing
api io_uring_prep_read_fixed() [1]) takes a buf_index, not an address.
Userspace would pass the ent->zero_copy_index as the buf index.
ent->zero_copy_index corresponds to the fixed buffer where userspace
told the kernel (at sqe registration time through the "uint16_t
zero_copy_index" init param) to place the zero-copied pages for that
sqe.

>
> This series adds buffer pools (implemented internally as a buffer ring) and
> zero-copy to fuse io-uring.
>
> Prior to this series, ents and buffers are tightly coupled where each entry
> has its own dedicated payload buffer, requiring N buffers for N entries where
> each buffer must be large enough to accomodate the maximum payload size. This
> is suboptimal as most request types require vastly less bytes than the maximum
> payload size and some requests do not require payload buffers at all.
>
> This series allows servers to pass in a buffer pool (internally managed as a
> buffer ring) that the kernel will use for servicing ents/requests. This
> decoupling reduces the memory usage requirements needed to use fuse-io-uring.
> It allows for future optimizations such as incremental buffer consumption and
> supporting multiple buffer sizes per ring.
>
> This series also allows servers to use io-uring registered buffers as the
> buffer pool which eliminates the overhead of pinning/unpinning user pages and
> translating virtual addreses per i/o request.
>
> This series additionally adds zero copy to fuse io-uring. The server can
> directly access client pages or page cache folios without copying data through
> an intermediary buffer. This requires CAP_SYS_ADMIN privileges and using
> registered buffers. Zero copy (patch 9) has a dependency on io-uring registered
> bvec changes in [1].

>
> This series is on top of commit f21c8b1a3b740 in for-next.
>
> The throughput improvements from registered buffers and zero-copy depends on
> how much of the server's per-request latency is spent on data copying vs
> backing I/O. When backing I/O dominates, the saved memcpy is a negligible
> fraction of overall latency. Please also note that for the server to
> read/write into the zero-copied pages, the read/write must go through io-uring
> as an IORING_OP_READ_FIXED / IORING_OP_WRITE_FIXED operation. If the
> server's backing I/O is instantaneous (eg served from cache), the
> overhead of the additional io_uring operation may negate the savings
> from eliminating the memcpy.
>
> In benchmarks using passthrough_hp on a high-performance NVMe-backed
> system, using registered buffers showed around a 10% throughput improvement
> for direct randreads (~2150 MiB/s to ~2400 MiB/s), a 4% improvement for direct
> sequential reads (~2510 MiB/s to ~2620 MiB/s), a 8% improvement for buffered
> randreads (~2100 MiB/s to ~2280 MiB/s), and a 6% improvement for buffered
> sequential reads (~2500 MiB/s to ~2670 MiB/s).
>
> Zero-copy showed around a 35% throughput improvement for direct
> randreads (~2150 MiB/s to ~2900 MiB/s), a 15% improvement for direct
> sequential reads (~2510 MiB/s to ~2900 MiB/s), a 15% improvement for
> buffered randreads (~2100 MiB/s to ~2470 MiB/s), and a 10% improvement
> for buffered sequential reads (~2500 MiB/s to ~2750 MiB/s). I didn't see
> enough of a clear improvement for writes due to write latency being I/O
> dominated.
>
> The benchmarks were run using:
>   fio --name=test_run --ioengine=sync --rw=rand{read,write} --bs=1M
>   --size=1G --numjobs=2 --ramp_time=30 --group_reporting=1
>
> The libfuse changes can be found in [2]. To test the server, run:
> sudo ~/libfuse/build/example/passthrough_hp ~/src ~/mounts/tmp
> --nopassthrough -o io_uring_zero_copy -o io_uring_q_depth=8
> Once this series is merged, the libfuse changes will be tidied up and
> submitted upstream.
