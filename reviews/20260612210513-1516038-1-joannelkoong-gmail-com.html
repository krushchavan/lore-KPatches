Patch summary
This patch series adds buffer pools and zero-copy to the fuse io-uring implementation, allowing servers to pass in a managed buffer pool that reduces memory usage requirements and enables future optimizations. The decoupling of ents and buffers also eliminates the need for dedicated payload buffers, reducing memory waste. Additionally, the series introduces zero-copy functionality, which allows direct access to client pages or page cache folios without copying data through an intermediary buffer, requiring CAP_SYS_ADMIN privileges and registered buffers.

The author addressed a concern about the tight coupling between ents and buffers in fuse io-uring, explaining that they have decoupled them by allowing servers to pass in a buffer pool (a contiguous allocation of buffers) that the kernel will use for servicing ents/requests. Show original comment 
Right now, ents and buffers are tightly coupled in fuse io-uring where
each entry has its own dedicated payload buffer, requiring N buffers for
N entries where each buffer must be large enough to accomodate the
maximum payload size. This is suboptimal as most request types (lookup,
open, release, getattr, etc) require vastly less bytes than the maximum
payload size and some requests (unlink, rmdir, fsync, flush, etc) do not
require payload buffers at all.

Instead of requiring a 1:1 coupling between ents and payload buffers,
allow the server to pass in a buffer pool (a contiguous allocation of
buffers) that the kernel will use as it wishes for servicing
ents/requests. Buffers are now pooled and selected on demand. Entries
only "reserve" a buffer while actively processing a request that
requires a payload buffer. This decoupling reduces the memory usage
requirements needed to use fuse-io-uring. It allows for future
optimizations such as incremental buffer consumption which will allow
non-overlapping regions of a single buffer to be used simultaneously
across multiple ents/requests, and the possibility of multiple buffer
sizes to be supported per ring in the future.

To use this, the server must set the FUSE_URING_BUFPOOL flag during
registration and provide the payload buffer pool through the iovec
payload in the first sqe. The server must set init.buf_size and
init.nr_bufs in the struct fuse_uring_cmd_req it passes to the kernel to
describe the buffer pool configuration. Buffer pool usage is set on a
per-ring basis. All subsequent registration SQEs for the same ring must
pass in consistent flags.

Internally, a buffer ring data structure is used for managing the buffer
pool. The kernel manages buffer selection and recycling through this
ring structure. A payload buffer is selected from the ring on demand if
the request requires a payload buffer and recycled after the request
completes if used.

The contiguous buffer pool passed in is partitioned into individual
buffers according to the buf_size passed in at registration time.

  PAYLOAD BUFFER POOL (contiguous, provided via iov[1] in first sqe):
    |<-------------- payload_size ------------>|
    +--------- --+-----------+-----------+-----+
    |  buf [0]   |  buf [1]  |  buf [2]  | ... |
    |  buf_size  |  buf_size |  buf_size | ... |
    +--------- --+-----------+-----------+-----+

The buffer ring logic is as follows:
select:  buf = bufs[head % nbufs]; head++
recycle: bufs[tail % nbufs] = buf; tail++
empty:   tail == head (no buffers available)
full:    tail - head >= nbufs

Allow servers to use an io-uring registered buffer as the buffer pool.
When the server registers buffers with io-uring, the pages backing the
buffers are pinned upfront. This eliminates the overhead of
pinning/unpinning user pages and translating virtual addresses per i/o
request. This also allows servers to use the same registered buffers for
subsequent backing store I/O (eg read_fixed/write_fixed), keeping data
in the same pinned pages without additional pinning or mapping overhead
required.

enabled. To use registered buffers, the server needs to register the
payload buffers through io-uring and during registration, set the
FUSE_URING_REGISTERED_BUFFERS init flag (alongside the
FUSE_URING_BUFPOOL flag) in the registration sqe. For all sqes the
server passes in (including the registration ones), it needs to follow
standard protocol for using registered buffers (eg set
sqe->uring_cmd_flags to include IORING_URING_CMD_FIXED, and pass in the
index where the registered buffers reside to sqe->buf_index). Kernel
support for registered buffers is advertised to userspace via the
FUSE_HAS_IO_URING_BUFPOOL init flag (a single umbrella flag).

system, using registered buffers showed around a 10% throughput
improvement for direct randreads (~2150 MiB/s to ~2400 MiB/s), a 4%
improvement for direct sequential reads (~2510 MiB/s to ~2620 MiB/s),
a 8% improvement for buffered randreads (~2100 MiB/s to ~2280 MiB/s),

Implement zero-copy for fuse io-uring, eliminating memory copies between
userspace, the kernel, and the fuse server for read/write operations.
The server can directly access client pages or page cache folios without
copying data through an intermediary buffer. When a fuse request arrives,
the kernel registers the relevant pages into a sparse slot in the
server's io_uring registered buffer table. The server can then operate
on these pages directly using io-uring fixed buffer operations (eg
read_fixed/write_fixed) and the kernel unregisters these pages when the
request completes. Non-page-backed args (eg op out headers) will go
through the payload buffer as normal.

This requires CAP_SYS_ADMIN and FUSE_URING_REGISTERED_BUFFERS (which
requires using bufpools). Gating on registered buffers keeps the
configuration space small and avoids partially-optimized modes that are
unlikely to be useful in practice. This is gated behind CAP_SYS_ADMIN
because zero-copy allows the server direct access to the client's
underlying pages, rather than operating on an intermediary buffer that
the contents of the client's pages were copied into or on page cache
folios. Kernel support for zero copy is advertised to userspace via the
FUSE_HAS_IO_URING_BUFPOOL init flag (a single umbrella flag).

The zero-copy read path is analogous.

Some requests may have both page-backed args and non-page-backed args.
For these requests, the page-backed args are zero-copied while the
non-page-backed args are copied to the buffer selected from the buffer
ring:
    zero-copy: pages registered via io_buffer_register_bvec()
    non-page-backed: copied to payload buffer via fuse_copy_args()

For a request whose payload is zero-copied, the
registration/unregistration path looks like:

    register:  fuse_uring_set_up_zero_copy()
                 folio_get() for each folio
                 io_buffer_register_bvec(ent->zero_copy_index)

    unregister: fuse_uring_req_end()
                  io_buffer_unregister(ent->zero_copy_index)
                  -> fuse_zero_copy_release() callback
                     folio_put() for each folio

Please note that on abort for in-flight zero-copied requests that have
been sent to userspace, the registered bvec slot remains occupied and
its folios remain pinned until the io-uring ring is destroyed, at which
point io-uring unregisters all buffers and the fuse_zero_copy_release()
callback drops the folio references. Unregistering at teardown would
require operating on the ring context directly, whose validity is hard
to ascertain; this is deemed not worth the complexity for the abort
race, since everything is freed when the ring is torn down.

The throughput improvement from zero-copy depends on how much of the
per-request latency is spent on data copying vs backing I/O. When
backing I/O dominates, the saved memcpy is a negligible fraction of
overall latency. Please also note that for the server to read/write
into the zero-copied pages, the read/write must go through io-uring
as an IORING_OP_READ_FIXED / IORING_OP_WRITE_FIXED operation. If the
server's backing I/O is instantaneous (eg served from cache), the
overhead of the additional io_uring operation may negate the savings
from eliminating the memcpy.

In benchmarks using passthrough_hp on a high-performance NVMe-backed
system, zero-copy showed around a 35% throughput improvement for direct
randreads (~2150 MiB/s to ~2900 MiB/s), a 15% improvement for direct
sequential reads (~2510 MiB/s to ~2900 MiB/s), a 15% improvement for
buffered randreads (~2100 MiB/s to ~2470 MiB/s), and a 10% improvement
for buffered sequential reads (~2500 MiB/s to ~2750 MiB/s).

The benchmarks were run using:
  fio --name=test_run --ioengine=sync --rw=rand{read,write} --bs=1M
  --size=1G --numjobs=2 --ramp_time=30 --group_reporting=1
