[PATCH v3 03/12] net: change sock.sk_ino and sock_i_ino() to u64

View on lore.kernel.org →
Active on: 2026-03-04
Patch summary
This patch updates the Linux kernel to use a 64-bit unsigned integer (u64) instead of a 32-bit unsigned long for storing inode numbers in socket structures and functions. This change is necessary because inode->i_ino is being converted to u64, and using an unsigned long would cause truncation on 32-bit architectures. The patch updates various files to reflect this change, including sock.h, net/ax25/af_ax25.c, and others. It also fixes format strings that print the result of sock_i_ino() to use %llu instead of %lu. The UAPI socket diag structures are left unchanged as they cannot be modified without breaking the ABI.
Jeff Linux-f2fs-devel 2026-03-04 Neutral LLM
The reviewer noted that the patch changes sock.sk_ino from unsigned long to u64, but does not update the format string in net/can/bcm.c to match the new type. The reviewer suggested updating the format string to %llu.
Show original comment 
inode->i_ino is being converted to a u64. sock.sk_ino (which caches the
inode number) must also be widened to avoid truncation on 32-bit
architectures where unsigned long is only 32 bits.

Change sk_ino from unsigned long to u64, and update the return type
of sock_i_ino() to match. Fix all format strings that print the
result of sock_i_ino() (%lu -> %llu), and widen the intermediate
variables and function parameters in the diag modules that were
using int to hold the inode number.

Note that the UAPI socket diag structures (inet_diag_msg.idiag_inode,
unix_diag_msg.udiag_ino, etc.) are all __u32 and cannot be changed
without breaking the ABI. The assignments to those fields will
silently truncate, which is the existing behavior.
