[PATCH] mm: don't allow empty relative nodemask in mpol_relative_nodemask()

Active on: 2026-05-29
Patch summary
This patch prevents empty relative nodemasks in mpol_relative_nodemask() by adding a check for the node weight being zero, which is not sufficient as it allows empty nodemasks for certain policy modes. Instead, the full policy-nodemask checking section from mpol_new() should be called from mpol_set_nodemask(). This ensures that all places where mpol_relative_nodemask() is called perform the same validation as mpol_new().
Joshua Hahn (author) 2026-05-29 Needs Work LLM
Author acknowledged that an empty nodemask should be checked for in mpol_relative_nodemask(), but disagreed with the proposed solution of checking if node weight is 0, instead suggesting to restructure and call the full policy-nodemask checking section from mpol_new(). Author acknowledges that mpol_set_nodemask should never be called with an empty nodemask if MPOL_F_RELATIVE_NODES is set, and suggests additional checks in mpol_rebind_policy to catch other potential failures. Author acknowledged a mistake in their original patch, specifically that they missed the case where MPOL_DEFAULT and MPOL_PREFERRED allow empty nodemasks when !STATIC && !RELATIVE, and agreed to restructure the code to fix this issue. Author acknowledges that the proposed patch approach won't work for MPOL_DEFAULT and MPOL_PREFERRED cases, agrees that mpol_new's policy-nodemask checking section should be called from mpol_set_nodemask as well.

Hi Joshua.

Indeed, quick and dirty shot.

The problem is that nodes_fold() can't work with the sz == 0. In
other words, folding to a 0-bit bitmap is an error. We don't check
that on bitmaps level because it's an internal helper, and it's a
caller's responsibility to validate the parameters.

nodes_onto(), or more specifically bitmap_onto(), is a different
story. In case of empty relmap, the function actually clears all the
bits in dst and returns.

I see 2 options to move this forward.

1. Simply disallow empty relmap in mpol_relative_nodemask(). There's
no valid cases for it, AFAIK, so the nodes_fold() limitation looks
reasonable. We can consider it as a new policy.

We've got 2 users for mpol_relative_nodemask(). In mpol_set_nodemask()
we can simply propagate the error; and in mpol_rebind_nodemask() we
can throw a warning and do nothing.

2. Follow the spirit of the nodes_onto(), and in case of empty
relmask, clean the ret mask and bail out

I'm in a favor for the 1st option, because empty relmask looks buggy
anyways.
