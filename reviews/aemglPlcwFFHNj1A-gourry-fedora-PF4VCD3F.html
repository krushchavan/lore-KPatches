[PATCH RFC v3 01/19] mm: thread user_addr through page allocator for cache-friendly zeroing

Active on: 2026-04-24 • 2026-04-23
Patch summary
This patch proposes a change to how the page allocator handles zeroing of memory, specifically for cache-friendly zeroing. It involves passing user addresses through the page allocator and using a wrapper function to handle cacheline alignment on the way out. However, after exploring this approach, it was deemed that the increased complexity may not be worth the benefits.

    old_interface(..., user_addr);

or

    __internal(..., user_addr) { }

    old_interface(...) {
       __internal(..., NO_USER_ADDR);
    }

    new_interface(..., user_addr) {
       __internal(..., user_addr);
    }

    /* 
     * Update some callsites to use new_interface()
     * mostly a bunch of _mpol() functions but some others
     */

    old_interface();
    /* don't fall folio_zero_user() - Bug! */

There is no elegant fix here if we want to do this all inside the buddy.

If the performance increase weren't so blatantly dramatic we could also
consider just not doing this at all.

There's maybe an alternative here in considering a change to the way the
