Patch summary
This patch adds a new function called get_user_page_vma() to the Linux kernel's memory management system. It is designed to fault in a single page under a held lock, either the mmap lock or the per-VMA lock. The function is intended for use by __access_remote_vm(), which needs to access a page that has already been looked up and locked. The patch includes several changes to existing functions, including get_gate_page() and faultin_page(), to support this new functionality. It also introduces a new flag called FOLL_VMA_LOCK to indicate when the per-VMA lock is being used. The patch aims to simplify the process of accessing pages under a held lock, making it easier for kernel developers to write efficient and safe code.

Author acknowledged that the current behavior of falling back to ->access() hook is correct, even though the implementation needs improvement. They proposed changing check_vma_flags() to ignore specified flags, specifically VM_IO | VM_PFNMAP, to achieve this. Author acknowledged a potential issue on systems with virtually indexed caches and expressed interest in restructuring get_user_page_vma_remote() to avoid double VMA lookup.
