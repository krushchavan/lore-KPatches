[f2fs-dev] [PATCH v2 17/18] iomap: pass iomap_next_fn directly instead of struct iomap_ops

Submitted: 2026-06-30  ·  Last comment: 2026-07-02

Patch summary
This patch simplifies the iomap API by removing the struct iomap_ops and passing the iomap_next function directly to various functions. This change is possible because all filesystems have implemented the ->iomap_next() method, eliminating the need for the legacy fallback methods. The patch makes no functional changes but reduces code complexity and improves maintainability.
Joanne Koong (author) 2026-06-30 Neutral LLM
Author acknowledged that the swapoff path needs to drop the per-vswap spinlock before calling try_to_unmap(), agreed to restructure in v2.
Show original comment 
Now that all filesystems implement ->iomap_next() and the legacy
->iomap_begin()/->iomap_end() fallback is gone, struct iomap_ops only
wraps a single iomap_next function pointer. Drop the struct entirely and
pass the iomap_next_fn directly to iomap_iter() and all the iomap/dax
entry points; filesystems pass their ->iomap_next function instead of an
ops struct.

No functional change intended.
