[PATCH 10/10] nfsd: validate symlink target length in NFSv4 CREATE

View on lore.kernel.org →
Submitted: 2026-05-29
Active on: 2026-05-29
Patch summary
This patch series addresses a security vulnerability in the NFSv4 protocol by validating the length of symlink targets in NFSv4 CREATE operations. The current implementation allows an unbounded allocation of up to 1 MiB per COMPOUND operation, which can lead to a double-put bug. The patch rejects oversized targets with ENAMETOOLONG and bounds the allocation.
Chuck Lever 2026-05-29 Reviewed-by Neutral LLM
Suggested using NFS4_MAXPATHLEN instead of PATH_MAX to express the protocol-layer intent and match the cross-version idiom. Also pointed out that the new boundary condition differs from v2/v3, recommending a change to '> NFS4_MAXPATHLEN' to align with v2/v3.

Nit: The protocol already has NFS4_MAXPATHLEN defined to PATH_MAX.
The v3 decoder uses its analog NFS3_MAXPATHLEN. Using
NFS4_MAXPATHLEN here expresses the protocol-layer intent and
matches the cross-version idiom.

The new boundary condition differs from v2/v3. v3 uses
"tlen > NFS3_MAXPATHLEN", accepting a target of exactly PATH_MAX
bytes; this uses ">= PATH_MAX", rejecting it. Switching to 
"> NFS4_MAXPATHLEN" both adopts the named constant and realigns
the maximum accepted length with v2/v3.
