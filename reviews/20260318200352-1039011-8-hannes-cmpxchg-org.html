Patch summary
This patch switches the deferred split shrinker to use list_lru, which properly sets up per-node and per-cgroup lists. This improves handling of cgroups on NUMA systems by avoiding splitting large pages on other nodes when a node-restricted allocation is reclaiming memory. The change also streamlines list operations and reclaim walks, and is consistent with how other major shrinkers use list_lru.

The reviewer pointed out that the deferred split shrinker's list_lru allocation is not properly synchronized, and suggested making the code more concise. The reviewer pointed out that the patch's explanation of the deferred split shrinker's call flow is unclear and suggested a more accurate representation by tracing through the function calls from shrink_slab_memcg() to deferred_split_scan(). The reviewer noted that the deferred split queue uses a per-memcg list, but also sets a bit in shrinker_info->unit->map, which is used to determine the shrink_control in shrink_slab_memcg(), potentially leading to confusion or incorrect behavior. The reviewer noted that the deferred_split_scan() function walks the same memcg->split_queue regardless of node, which they consider suboptimal. Reviewer Lorenzo suggested exposing deferred_split_lru as a getter or creating a new function folio_memcg_alloc_deferred to encapsulate the allocation of the list lru, allowing callers outside huge_memory.c to access it without having to use the global variable. Reviewer Lorenzo suggested splitting the huge zero folio handling into its own function, but noted it's out of scope for this series and should be addressed in a future patch. The reviewer questioned whether there's a potential race condition when the shrinker is freed, causing the deferred_split_lru to be accessed by another thread. Reviewer Lorenzo suggested renaming the 'lru' variable to something more descriptive, citing that single-letter variable names are not conventional Reviewer Lorenzo questioned why the deferred split queue was switched to list_lru for anonymous pages specifically, indicating a lack of understanding of the change and requesting clarification. Reviewer Lorenzo expressed concern about a hidden RCU lock in the code and suggested refactoring the function, possibly splitting out folio_ref_freeze() bits into a separate helper function. Reviewer Lorenzo pointed out a minor issue with variable naming conventions, specifically that single-letter variable names can be confusing and suggested that 'l' is particularly problematic due to its similarity in appearance to the digit 1. The reviewer suggested refactoring the code to avoid duplicated logic in list_lru_del(), specifically by making it reference _deferred_list explicitly, instead of having it hardcoded in multiple places. The reviewer questioned the need for holding a lock while calling list_lru_add(), suggesting that using a locked_add_variant would be more intuitive and avoiding the need for irqsave/restore. Reviewer Lorenzo agrees that the partially mapped logic is correct, as folio_try_get() will only fail to add a reference if the folio has already lost the race. The reviewer suggests adding an irqsave/restore variant of list_lru_add() to handle the repeating pattern of acquiring and releasing locks in interrupt-safe manner. Reviewer Lorenzo suggested moving a stat counter increment operation underneath an if statement to prevent it from being incremented when the operation fails. The reviewer suggests refactoring the large if-branch in deferred_split_folio() into a separate patch for better bisectability and reviewability. The reviewer expressed appreciation for the patch's code deletion and thanked the submitter, indicating a positive sentiment.
