Patch summary
This patch series adds buffer pools and zero-copy functionality to the fuse io-uring implementation, allowing servers to pass in a managed buffer pool that reduces memory usage requirements and enables future optimizations. It also allows for direct access to client pages or page cache folios without copying data through an intermediary buffer, requiring CAP_SYS_ADMIN privileges. The changes are based on registered buffers and zero-copy, which can provide significant throughput improvements depending on the server's per-request latency and backing I/O characteristics.

The author is addressing feedback on the buffer allocation mechanism in fuse io-uring, specifically the issue that most request types require less than the maximum payload size, leading to inefficient memory usage. Miklos,

This series has stalled on an architectural question and I'd
appreciate your input on the direction.

Background:
In fuse io-uring, entries are reusable communication slots between the
kernel and the daemon. When a request needs to be sent from the kernel
to the daemon, it is assigned to an available entry. The entry carries
the request to userspace, and the response comes back through the same
entry. The number of entries determines how many requests can be
serviced concurrently. Currently, when the daemon registers an entry
during fuse io-uring setup, it passes in a payload buffer (sized to
the maximum payload, typically 1 MiB) alongside it. The buffer is tied
to the entry for the lifetime of the connection.

Problem that needs to be solved:
Most request types (lookup, getattr, open, etc) require far less than
the maximum payload size, and some (flush, fsync, etc) require no
payload buffer at all. Every entry reserves a full 1 MiB buffer
regardless of actual need. On systems with many entries, this memory
cost scales linearly - for example, 64 entries across 8 queues
requires 512 MiB of payload buffers alone.

Approach A:
Decouple entries from payload buffers. Entries become generic - any
entry can use any buffer from the pool. At setup time, the daemon
passes in a contiguous pool of memory, and the kernel dynamically
selects and recycles buffers from this pool as entries are assigned
requests. Entries servicing requests with no payloads consume no
buffer, and the pool can have fewer buffers than entries. This
supports adding multi-size buffers and incremental buffer consumption
(multiple entries using non-overlapping regions of the same buffer
concurrently) as optimizations in the future.

Approach B:
Keep the 1:1 entry-buffer coupling, but allow entries to be registered
with different payload buffer sizes at setup time. The daemon would
register entries with smaller buffers for metadata-heavy workloads and
larger buffers for data-heavy ones. This keeps the existing model of
passing in a payload buffer during entry registration at setup time
and requires no uapi additions. Each entry registration adds buffer
capacity whereas with buffer pools, buffer capacity would have to be
added as a separate pool.

I want to make sure I'm representing both approaches fairly - Bernd,
please correct anything I've gotten wrong.

Miklos, what are your thoughts?

Thanks,
Joanne

> N entries where each buffer must be large enough to accomodate the
> maximum payload size. This is suboptimal as most request types (lookup,
> open, release, getattr, etc) require vastly less bytes than the maximum
> payload size and some requests (unlink, rmdir, fsync, flush, etc) do not
> require payload buffers at all.
>
> Instead of requiring a 1:1 coupling between ents and payload buffers,
> allow the server to pass in a buffer pool (a contiguous allocation of
> buffers) that the kernel will use as it wishes for servicing
> ents/requests. Buffers are now pooled and selected on demand. Entries
> only "reserve" a buffer while actively processing a request that
> requires a payload buffer. This decoupling reduces the memory usage
> requirements needed to use fuse-io-uring. It allows for future
> optimizations such as incremental buffer consumption which will allow
> non-overlapping regions of a single buffer to be used simultaneously
> across multiple ents/requests, and the possibility of multiple buffer
> sizes to be supported per ring in the future.
>
> To use this, the server must set the FUSE_URING_BUFPOOL flag during
> registration and provide the payload buffer pool through the iovec
> payload in the first sqe. The server must set init.buf_size and
> init.nr_bufs in the struct fuse_uring_cmd_req it passes to the kernel to
> describe the buffer pool configuration. Buffer pool usage is set on a
> per-ring basis. All subsequent registration SQEs for the same ring must
> pass in consistent flags.
>
> Internally, a buffer ring data structure is used for managing the buffer
> pool. The kernel manages buffer selection and recycling through this
> ring structure. A payload buffer is selected from the ring on demand if
> the request requires a payload buffer and recycled after the request
> completes if used.
>
> The contiguous buffer pool passed in is partitioned into individual
> buffers according to the buf_size passed in at registration time.
>
>   PAYLOAD BUFFER POOL (contiguous, provided via iov[1] in first sqe):
>     |<-------------- payload_size ------------>|
>     +--------- --+-----------+-----------+-----+
>     |  buf [0]   |  buf [1]  |  buf [2]  | ... |
>     |  buf_size  |  buf_size |  buf_size | ... |
>     +--------- --+-----------+-----------+-----+
>
> The buffer ring logic is as follows:
> select:  buf = bufs[head % nbufs]; head++
> recycle: bufs[tail % nbufs] = buf; tail++
> empty:   tail == head (no buffers available)
> full:    tail - head >= nbufs

> +pinned upfront. This eliminates the overhead of pinning/unpinning user pages
> +and translating virtual addresses per i/o request. This also allows servers to
> +use the same registered buffers for subsequent backing store I/O (eg
> +read_fixed/write_fixed), keeping data in the same pinned pages without
> +additional pinning or mapping overhead required.

> +Registered buffers require the buffer pool (FUSE_URING_BUFPOOL) to be

> +payload buffers through io-uring and during registration, set the

> +Zero-copy

> +
> +Fuse io-uring zero-copy allows the server to directly read from / write to
