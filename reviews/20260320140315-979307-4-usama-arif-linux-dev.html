[PATCH v2 3/4] elf: align ET_DYN base to max folio size for PTE coalescing

Patch summary
This patch aligns the base of PIE binaries (ET_DYN) to the maximum folio size supported by the page cache, enabling hardware PTE coalescing and PMD mappings for large folios. The alignment is determined by considering the maximum folio size and segment size, ensuring that file-offset-aligned folios map to properly aligned virtual addresses. This fix addresses an issue where readahead patches cannot benefit from contpte mapping due to improper virtual address alignment.
Usama Arif (author) 2026-03-20 Neutral LLM
The author is addressing a concern about the alignment of PT_LOAD segments in ELF binaries, specifically for PIE (Position-Independent Executable) binaries. The reviewer had pointed out that the current implementation does not properly align the virtual address to the maximum folio size supported by the page cache, which can prevent hardware PTE coalescing and PMD mappings for large folios. The author has explained their reasoning behind the changes they made in the patch, including the use of the largest power-of-2 that fits within the segment size, capped by what the page cache will allocate. They have also clarified that the segment size check is used to avoid reducing ASLR entropy for small binaries. The author's response suggests that the patch will need further revision to address the reviewer's concerns.
Show original comment 
For PIE binaries (ET_DYN), the load address is randomized at PAGE_SIZE
granularity via arch_mmap_rnd(). On arm64 with 64K base pages, this
means the binary is 64K-aligned, but contpte mapping requires 2M
(CONT_PTE_SIZE) alignment.

Without proper virtual address alignment, readahead patches that
allocate 2M folios with 2M-aligned file offsets and physical addresses
cannot benefit from contpte mapping, as the contpte fold check in
contpte_set_ptes() requires the virtual address to be CONT_PTE_SIZE-
aligned.

Fix this by extending maximum_alignment() to consider the maximum folio
size supported by the page cache (via mapping_max_folio_size()). For
each PT_LOAD segment, the alignment is bumped to the largest
power-of-2 that fits within the segment size, capped by the max folio
size the filesystem will allocate, if:

  - Both p_vaddr and p_offset are aligned to that size
  - The segment is large enough (p_filesz >= size)

This ensures load_bias is folio-aligned so that file-offset-aligned
folios map to properly aligned virtual addresses, enabling hardware PTE
coalescing (e.g. arm64 contpte) and PMD mappings for large folios.

The segment size check avoids reducing ASLR entropy for small binaries
that cannot benefit from large folio alignment.
