[PATCH 2/3] mm/pagewalk: let folio_walk_start() run under the per-VMA lock

View on lore.kernel.org →
Submitted: 2026-06-25
Active on: 2026-06-25
Patch summary
This patch allows callers of folio_walk_start() to run under the per-VMA lock instead of the mmap lock, reducing contention and improving performance for readers that only need to access a single page. The patch adds a new flag FW_VMA_LOCKED, which requires CONFIG_MMU_GATHER_RCU_TABLE_FREE and excludes hugetlb VMAs. Callers must keep interrupts disabled until folio_walk_end(). This change does not affect existing callers, as none currently pass the FW_VMA_LOCKED flag.
Rik Riel (author) 2026-06-25 Neutral LLM
The author acknowledges a potential issue with khugepaged's handling of page tables during THP collapse, noting that relying solely on the RCU read lock could result in seeing old page table contents that no longer point to current process memory. The author suggests two possible solutions for khugepaged and asks if they are missing something.

One issue here is that while we can safely read
the old page table under the RCU read lock, in
the middle of a THP collapse there is no guarantee
that the old page table points at the process's 
current memory.

Khugepaged could fix this in one of two ways:
- zap all readers with an IPI, and use that as
  synchronization
- make sure the old page table's PTEs point at
  the individual pages inside the new PMD

Right now khugepaged does the first.

Relying only on the RCU read lock to read the
page table could result in us seeing old page
table contents, that no longer point at the
current process memory.

Lorenzo Stoakes 2026-06-25 Reviewed-by Neutral LLM
Reviewer Lorenzo Stoakes expressed frustration that the patch author did not respond to their review, and requested a response. The reviewer pointed out that the patch's description of munmap() behavior is incorrect, as it acquires the VMA lock and only removes upper-level tables if the entire range is spanned. They also stated that PTE page tables can be removed via MADV_DONTNEED under RCU, making the GUP fast imitating mechanism unnecessary. Reviewer noted that the patch's assumption about CONFIG_MMU_GATHER_RCU_TABLE_FREE is incorrect, as retract_page_tables() and try_collapse_pte_mapped_thp() use pte_free_defer(), which uses RCU instead of serializing against lockless walkers. Reviewer questioned the validity of adding FW_VMA_LOCKED flag, suggesting that they may be misunderstanding the patch's purpose or that there might be an issue with the locking mechanism. The reviewer found the patch description to be overly verbose and requested that it be edited for readability
