Reviewer Michal Hocko disagrees with the patch's approach, stating that the two implementations (traditional LRU and MGLRU) are not in feature parity or overlapping in bug space, and therefore should not be unified. Reviewer noted that MGLRU has multiple entry points in the shared node/memcg scanning code, which may create complexity and make it harder to maintain. Reviewer Michal Hocko suggested focusing on making a single canonical reclaim implementation work well, addressing most or all known regressions of MGLRU, and emphasized the importance of extensibility to support creative aging algorithms. Reviewer Michal Hocko expressed skepticism about unifying memory reclaim algorithms into a single pluginable code path, suggesting it should only be considered after MGLRU unification is complete and its potential has been fully explored.

Show original comment 
Hi Shakeel,

The reclaim_ops direction looks very promising. I'd be interested in the discussion.

We are particularly interested in the individual effects of several mechanisms
currently bundled in MGLRU. reclaim_ops would provide a great opportunity to
run ablation experiments, e.g. testing traditional LRU with page table scanning.

On policy granularity, it would also be interesting to see something like ``reclaim_ext''[1,2]
taking control at different levels, similar to what sched_ext does for scheduling policies.

Best,
Zicheng

[1] cache_ext: Customizing the Page Cache with eBPF
[2] PageFlex: Flexible and Efficient User-space Delegation of Linux Paging Policies with eBPF
