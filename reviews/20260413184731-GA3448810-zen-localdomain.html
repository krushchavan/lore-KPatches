Patch summary
This patch addresses a quota-related issue in the BTRFS file system, specifically an error that occurs when trying to free extents while running delayed refs. The problem arises from an invalid state caused by a previous transaction commit, leading to a '-2 No such entry' error. To fix this, the patch modifies the __btrfs_free_extent function to handle this specific scenario and prevent the quota-related errors. The approach taken involves adding additional checks and handling for the invalid state, ensuring that the BTRFS file system can recover from such situations without crashing or entering a readonly mode.

Reviewer expressed concern about the safety of enabling quotas in Linux kernel version 6.12 due to potential issues causing filesystem corruption and unmountability, specifically mentioning a report from 'moremagic' where the filesystem was unable to be mounted read/write. Reviewer noted that the issue is specific to Raspberry Pi's custom kernel version 6.12 and suggested disabling quotas due to potential instability, as they cannot afford a crash on their large filesystem. Reviewer Marc MERLIN expressed confusion about the patch's story and asked for clarification on how to fix quota problems, but did not raise any specific technical concerns or suggestions. Reviewer noted that switching to 16k native filesystems on arm caused issues with btrfs send/receive, specifically breaking the ability to send from a 4k FS to a 16k FS. Reviewer Marc MERLIN reported crashes on multiple systems after enabling quotas, citing a different path abort in __btrfs_free_extent due to an invalid state, and requested confirmation that quotas are not safe, especially with 16k pages in the kernel but 4k in the filesystem. Reviewer Marc MERLIN expressed confusion about the issue, admitted being clueless about btrfs kernel code, and took a workaround by disabling quotas on all but one system to gather more data for correlation.
