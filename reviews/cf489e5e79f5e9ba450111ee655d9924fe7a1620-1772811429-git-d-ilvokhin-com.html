Patch summary
This patch updates the reserve_highatomic_pageblock() function in the Linux kernel's page allocation code to use a zone lock guard instead of explicit lock/unlock and goto out_unlock patterns. The zone lock guard is a new feature introduced by this patch, which provides automatic scope-based cleanup when exiting the locked region. This change simplifies the code and reduces the risk of errors due to forgotten unlocks or incorrect locking orders. The patch also updates the include/linux/mmzone_lock.h file to define the zone lock guard and its attributes.
Dan Williams 2026-03-12 Tested-by Needs Work LLM
Reviewer Dan Williams noted that the zone_lock_irqsave lock guard does not properly handle the case where reserve_highatomic_pageblock() is called from an interrupt context, and suggested using a different locking mechanism to avoid potential deadlocks. Reviewer Dan Williams suggested using __GUARD_IS_ERR() only when a better implementation is available, and confirmed the patch works fine in the CXL subsystem tests.

cleanup: Optimize guards

Andrew reported that a guard() conversion of zone_lock increased the
code size unnecessarily.

It turns out the unconditional __GUARD_IS_ERR() is to blame. As
explored earlier [1], __GUARD_IS_ERR(), similar to IS_ERR_OR_NULL(),
generates somewhat sub-optimal code.

However, looking at things again, it is possible to avoid doing the
__GUARD_IS_ERR() unconditionally. Revert the normal destructors to a
simple NULL test and only add the IS_ERR bit to COND guards.

This cures the reported overhead; as compiled by GCC-16:

post:	Total: Before=45299, After=45026, chg -0.60%
