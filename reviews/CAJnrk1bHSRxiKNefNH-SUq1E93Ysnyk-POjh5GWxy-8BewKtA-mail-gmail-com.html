[PATCH v6 3/3] fuse: add an implementation of open+getattr

View on lore.kernel.org →
Active on: 2026-03-03 • 2026-03-02
Patch summary
This patch modifies the fuse implementation to compound FUSE_GETATTR with FUSE_OPEN only when necessary, specifically when file attributes are invalid or O_APPEND is set and the file's last access time has changed since it was opened.

Horst Birthelmer 2026-03-03 Reviewed-by Needs Work LLM
Horst Birthelmer suggested adding a 'compound flag' to allow optional operations, which would enable sending open+getattr but only execute FUSE_OPEN if the fuse server doesn't support compounds, thus preserving current behavior and allowing for potential performance improvements. Reviewer questioned the assumption that file attributes are locally cached, suggesting that getattr should only be compounded to open when attributes are actually invalid
Show original comment 
OK, I will add a 'compound flag' for optional operations and don't
execute those, when the fuse server does not support compounds.

This way we can always send the open+getattr, but if the fuse server
does not process this as a compound (aka. it would be costly to do it), 
we only resend the FUSE_OPEN. Thus the current behavior does not change
and we can profit from fuse servers that actually have the possibility to
give us the attributes.

We can take a look at when to fetch the attributes in another context for
the other cases.
