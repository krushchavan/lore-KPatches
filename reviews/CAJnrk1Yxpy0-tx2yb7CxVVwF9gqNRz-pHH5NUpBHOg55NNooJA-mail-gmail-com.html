[RFC PATCH v3 6/8] fuse: implementation of lookup_handle+statx compound operation

View on lore.kernel.org →
Submitted: 2026-04-07
Active on: 2026-04-07
Patch summary
This patch introduces a new struct fuse_entry2_out to handle lookup and statx operations in the FUSE file system, which is necessary because the original struct fuse_entry_out includes attributes that are not needed for this operation. The compound operation with statx is required because the node ID is not known yet when performing the lookup, so it uses a workaround with FUSE_ROOT_ID to get around this issue. The patch aims to optimize performance by reducing the size of the response and improve code organization by separating metadata operations from directory operations.
Luis Henriques 2026-04-07 Reviewed-by Needs Work LLM
Reviewer questioned the necessity of a new struct fuse_entry2_out and suggested reusing struct fuse_entry_out instead, pointing out that it would simplify ABI and potentially reduce memory usage by ignoring unnecessary attr fields. Luis Henriques explained that the main reason for using statx instead of attr in the lookup_handle operation is to include additional attributes, such as btime, and noted that this approach was suggested by Miklos to simplify the interface and reduce complexity. He also mentioned that FUSE_ROOT_ID is used as a workaround for the node id in compound commands and may be required in other operations.

First of all, thanks a lot for looking into this patchset.  Much
appreciated!

The main reason for swapping the usage of attr by statx is that statx
includes some attributes that attr does not (e.g. btime).  And since I was
adding a new FUSE operation, it would be a good time for using statx
instead.  (Moreover, as new attributes may be added to statx in the
future, the benefits of using statx could eventually be even greater.)

This was suggested by Miklos here[0], before converting the whole thing to
use compound commands.  So, I was going to use fuse_statx in the _out args
for lookup_handle.  However, because the interface was getting a bit
complex with extra args (and ext headers!), Miklos ended up suggesting[1]
to remove attr completely from the lookup_handle operation, and use
compounds instead to have the full functionality.

Obviously, I may have misunderstood (or mis-implemented) the suggestions
that were done.  And hopefully the provided links to the discussion that
originated this approach will help.

Regarding the usage of FUSE_ROOT_ID as a workaround for the node id, I
believe this is a more generic problem which will occur in other compound
commands as well.  If we want to create a new file system object and
perform some operation with it within the same compound, a similar
workaround will be required (or some sort of flag in the compound command
to signal this dependency).

I hope this helped to clarify a bit your questions.
