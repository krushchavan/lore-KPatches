[PATCH v2 3/3] arm64: escalate smp_send_stop() to an SDEI NMI as a last resort

View on lore.kernel.org →
Submitted: 2026-06-10
Active on: 2026-06-10
Patch summary
This patch adds an additional escalation path to the smp_send_stop() function on arm64 architecture, which is used to stop a CPU that has become wedged and is ignoring interrupts. When the regular IPI (Inter-Processor Interrupt) and pseudo-NMI paths fail, the patch triggers an SDEI (System Description Event Interface) NMI (Non-Maskable Interrupt) as a last resort. This allows the system to recover from the wedged CPU by sending it an SDEI event that will eventually lead to its shutdown. The patch also introduces new functions and modifications to existing code to support this escalation path.
Doug Anderson 2026-06-10 Inline Review Reviewed-by Needs Work LLM
Reviewer Doug Anderson noted that the patch does not handle the case where the SDEI event is already pending when trying to escalate to it, and requested a check for this condition before triggering the SDEI event. Reviewer Doug Anderson suggested combining the stop functionality into a single function arm64_nmi_cpu_stop() to avoid synchronization issues across different stop paths, and provided an example implementation. Reviewer Doug Anderson suggested that the patch could be improved by separately testing for SDEI availability before attempting to use it, and proposed modifying sdei_nmi_stop_cpus() to return void instead of printing a message. Reviewer Doug Anderson questioned the shared EVENT ID for 'backtrace' and 'stop', suggesting separate IDs would improve code understandability and allow elimination of the sdei_nmi_stop_mask variable. Can we combine everything into one function so we don't have to keep
all the different stop functionality in sync? Like this (untested):

void __noreturn arm64_nmi_cpu_stop(struct pt_regs *regs, bool die_on_crash)
{
  unsigned int cpu = smp_processor_id();
  bool crash = IS_ENABLED(CONFIG_KEXEC_CORE) && crash_stop;

  /*
   * Use local_daif_mask() instead of local_irq_disable() to make sure
   * that pseudo-NMIs are disabled. The "stop" code starts with
   * an IRQ and falls back to NMI (which might be pseudo). If the IRQ
   * finally goes through right as we're timing out then the NMI could
   * interrupt us. It's better to prevent the NMI and let the IRQ
   * finish since the pt_regs will be better.
   */
  local_daif_mask();

  if (crash)
    crash_save_cpu(regs, cpu);

  set_cpu_online(cpu, false);

  sdei_mask_local_cpu();

  if (crash && die_on_crash)
    __cpu_try_die(cpu);

  /* just in case */
  cpu_park_loop();
}

Then in do_handle_IPI(), it's just:

  case IPI_CPU_STOP:
  case IPI_CPU_STOP_NMI:
    arm64_nmi_cpu_stop(get_irq_regs(), true);
    break;

...and from the SDEI code you pass "false" for "die_on_crash".

Perhaps when doing that you'd stop unconditionally getting the cpu in
do_handle_IPI() and just call it for `IPI_KGDB_ROUNDUP` since it would
now be the only consumer of that local variable.

FWIW, I'm not totally sure I followed the logic for why "die_on_crash"
needs to be "false" for the SDEI case, but I also am not terribly
familiar with KEXEC and crash-dumping kernels, since I've never
attempted to get that feature to work.
