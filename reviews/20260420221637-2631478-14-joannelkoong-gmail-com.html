[PATCH v1 13/17] fuse: add passthrough support for atomic file creation

Patch summary
This patch adds support for atomic file creation through the FUSE (Filesystem in Userspace) interface. It allows the server to return a backing ID, which is used to associate the newly created inode with the backing file for passthrough operations before the dentry is instantiated. The patch modifies the fuse_create_open function to use either the original entry_out structure or the new entry2_out structure depending on whether FUSE_PASSTHROUGH_INO is enabled. If entry2_out is used, it processes the server's response and extracts the backing ID. This change enables atomic file creation for FUSE filesystems that support passthrough operations.

Joanne Koong (author) 2026-04-20 Neutral LLM
Author acknowledged that enforcing FOPEN_PASSTHROUGH and backing id in FUSE_OPEN/FUSE_CREATE reply is overkill, agreed to remove this restriction in v2
Show original comment 
Currently in this series, the kernel enforces that if there is a
backing file registered for the inode, the server is still expected to
set FOPEN_PASSTHROUGH and provide the backing id in the FUSE_OPEN /
FUSE_CREATE reply. My original motivation was for keeping the api
contract consistent between FUSE_PASSTHROUGH_INO and the regular
passthrough mode, but I think you're right that this is just overkill.
I'll remove this restriction in v2.

Thanks,
Joanne
