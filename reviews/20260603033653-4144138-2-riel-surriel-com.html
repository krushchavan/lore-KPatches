Patch summary
This patch replaces the hand-rolled red-black tree in the IOVA allocator with a Maple tree, a B-tree variant designed for range tracking and gap searching. The change reduces memory usage per IOVA entry from around 64 bytes to around 44-57 bytes, while also improving search performance by reducing the search depth to O(log_10 n). The patch also simplifies the code by eliminating the need for retry loops and successor gap fixups.
Liam Howlett 2026-06-17 Inline Review Reviewed-by Neutral LLM
Reviewer Liam Howlett questioned the use of an external lock when the spinlock is intended to protect the tree, suggesting a potential inconsistency in locking behavior. Reviewer Liam Howlett noted that the alloc_fail label in the error recovery path may overwrite the max32_alloc_size, which is intended to ensure 32-bit safety, and requested that this be fixed. Reviewer Liam Howlett questioned the possibility of a specific alignment scenario occurring in the IOVA allocator, specifically that the worst-case alignment gap found by mas_empty_area_rev() could be smaller than both the start and end of the search range. Reviewer noted that the maple state can be used to avoid rewalking the tree when searching for specific slots, potentially reducing dereference overhead.
