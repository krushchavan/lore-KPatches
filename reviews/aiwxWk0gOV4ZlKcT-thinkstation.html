[PATCH v1] kasan: Fix false-positive wild-memory-access on x86 under 5-level paging

View on lore.kernel.org →
Submitted: 2026-06-12
Active on: 2026-06-12
Patch summary
This patch fixes a false-positive wild-memory-access error in the KASAN (Kernel Address Sanitizer) on x86 systems using 5-level paging. The issue occurs because the `cpu_feature_enabled()` function reads the live bits of CPU features before alternatives are patched, leading to incorrect results in interrupt context. To fix this, the patch modifies the `identify_cpu()` function to save and restore interrupts around the code that populates the `x86_capability` structure, ensuring that the correct values are read even in interrupt context. This change is specific to x86 systems and does not affect other architectures.
Ihor Solodrai 2026-06-12 Inline Review Reviewed-by Needs Work LLM
Reviewer Ihor Solodrai expressed uncertainty about the fix proposed by Kiryl Shutsemau, questioning whether disabling IRQs in identify_cpu() is a complete solution and potentially introducing undesirable consequences, such as NMI handling issues. Reviewer Ihor Solodrai expressed doubts about disabling IRQs in identify_cpu() as the solution to prevent a window where boot_cpu_data.x86_capability reads back zero, suggesting that this approach may not be correct because it only protects a portion of the capabilities array and could lead to issues with mcheck_cpu_init() which performs kmalloc(GFP_KERNEL) during this time. Reviewer Ihor Solodrai suggested defining a static key for LA57 feature in KASAN code to avoid performance issues and make the code x86-specific, but noted that this would not address the underlying issue of capabilities being temporarily cleared on boot.

Hi Kiryl, thank you for testing.

I thought of disabling IRQs here too, and AI bot flagged this as well [1].

I wasn't sure it'd be a good fix, because I don't know if it has any
undesirable consequences, and whether it's actually a complete fix.
For example, can a NMI hit this?

[1] https://lore.kernel.org/bpf/7b6d449f-c70c-4e8e-bfc4-a2f75517395c@linux.dev/

---

If we scope this narrowly, then it remains LA57-specific, because we protect
only a portion of the capabilities array.
IIUC to ensure x86_capability is "ready", a proper scope should be
entire identify_cpu(). And this may be a problem, because mcheck_cpu_init()
does kmalloc(GFP_KERNEL) [2], for example.

So I have doubts disabling IRQ is the right approach.

[2] https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/tree/arch/x86/kernel/cpu/mce/genpool.c?h=v7.1-rc7#n119

---

What if we define a static key for LA57 feature for KASAN?
This will avoid the performance hit we get with USE_EARLY_PGTABLE_L5.
And KASAN won't care about x86_capability being cleared.
Is that feasible/acceptable?

That will be a KASAN-specific LA57-specific fix however.
The underlying "capabilities are temporarily cleared on boot"
issue would remain. Although it wasn't a problem until this bug surfaced.
