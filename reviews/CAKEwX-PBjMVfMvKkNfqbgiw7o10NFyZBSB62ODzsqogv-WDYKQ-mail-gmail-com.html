Patch summary
This patch series aims to improve the performance of virtual swap space (vswap) by addressing a regression in handling zero-filled pages. The issue arises when memhog populates memory with 0xff, triggering full overhead for swapfile-backed swap entries even though they are 'same-filled'. The developer plans to handle non-zero same-filled pages and support this case in vswap without extra metadata overhead.
Kairui Song 2026-03-24 Inline Review Reviewed-by Needs Work LLM
Reviewer noted that same-filled pages are not necessarily zero-filled and requested support for non-zero same-filled pages in vswap, suggesting changing VSWAP_ZERO to VSWAP_SAME_FILLED and using the backend field to store the value. Reviewer Kairui Song noted that the patch series still exhibits performance issues even when swapping out zero-filled pages, and identified a critical issue with cgroup and global swap usage accounting being broken due to skipped allocations. Reviewer noted that the patch series' performance issue on zero-filled pages is likely due to memhog populating memory with 0xff, which triggers full swapfile-backed swap entry overhead, and suggested handling non-zero same-filled pages by changing the VSWAP_ZERO swap entry type to VSWAP_SAME_FILLED. Reviewer noted that the patch's performance issue is due to memhog populating memory with 0xff, which triggers full overhead of swapfile-backed swap entry even though it's 'same-filled' and not zero-filled.

thp_zero_page_alloc 0

swpin_zero 12239329

break that, that's one major reason why we use a zero entry instead of
mapping to a zero readonly page. If that is acceptable, we can have

That's still just an example. bypassing the accounting and still

performance and design.
