[PATCH] mm: khugepaged: free xarray nodes left behind by failed collapse_file()

View on lore.kernel.org →
Submitted: 2026-06-17
Active on: 2026-06-17
Patch summary
This patch addresses a memory leak issue in the khugepaged code, specifically when collapse_file() fails and leaves behind empty xarray nodes. These nodes are not freed on error paths, leading to their leakage until the inode is evicted. The patch introduces a mechanism to prune these empty nodes by storing an XA_RETRY_ENTRY into each slot and immediately clearing it, allowing the node and its ancestors to be freed.

That's not an issue.  The code which handles this is in
__filemap_add_folio().  You can see that we handle value (ie shadow)
entries here:

                xas_for_each_conflict(&xas, entry) {
                        old = entry;
                        if (!xa_is_value(entry)) {
                                xas_set_err(&xas, -EEXIST);
                                goto unlock;
                        }

(this loop will iterate 0 times if there are only empty nodes in the
"conflict" area)

Then we just do xas_store() which will overwrite the nodes.  That calls
xas_free_nodes() on any nodes which are overwritten.

Reviewer Jinjiang Tu noted that freeing empty xarray nodes in clear_inode() is sufficient, as a user cannot exhaust memory by continuously creating xa_node structures. Additionally, they suggested not to free these nodes in the collapse_file() rollback path, as it may lead to concurrent calls of collapse_file(), which creates new xa_nodes for the same region.
Show original comment 
A user cannot exhaust memory by continuously creating xa_node structures. So
it is enough to free emptyxa_nodes in clear_inode().

We shouldn't free empty xa_nodes in collapse_file() rollback path, because we
couldn't avoid a concurrent call of collapse_file(), which creates xa_nodes
for the same region and may use these nodes.
