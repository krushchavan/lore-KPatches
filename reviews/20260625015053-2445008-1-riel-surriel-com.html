Patch summary
This patch series introduces a fast path for __access_remote_vm that uses per-VMA locks instead of the process-wide mmap_lock, improving system monitoring reliability and speeding up the function under contention. The fast path is only enabled when it's safe to do so, requiring a kernel with per-VMA locks, RCU page table freeing, and specific VMA types excluded. This change aims to address issues where processes get stuck holding the mmap_lock for extended periods, hindering system monitoring tools' ability to assess and log situations.

One issue here is that while we can safely read
the old page table under the RCU read lock, in
the middle of a THP collapse there is no guarantee
that the old page table points at the process's 
current memory.

Khugepaged could fix this in one of two ways:
- zap all readers with an IPI, and use that as
  synchronization
- make sure the old page table's PTEs point at
  the individual pages inside the new PMD

Right now khugepaged does the first.

Relying only on the RCU read lock to read the
