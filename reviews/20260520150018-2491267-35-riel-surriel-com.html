[RFC PATCH 34/40] btrfs: allocate eb-attached btree pages as movable

View on lore.kernel.org →
Submitted: 2026-05-20
Active on: 2026-05-20
Patch summary
This patch modifies the Btrfs file system to allocate extent buffer pages attached to btree_inode as movable, allowing them to be migrated and reducing fragmentation in the page-superblock allocator. The change targets a dominant source of fragmentation observed on large test systems, where ~28 GB of btree_inode pages were parked across many tainted superpageblocks, preventing 1 GiB hugepage allocation from those regions. By adding the __GFP_MOVABLE flag to one allocation site and passing it through to other functions, the patch enables these pages to be moved to movable pageblocks, where they can be drained by the background defragger without needing a fallback LRU-sample approach.
Boris Burkov 2026-05-20 Inline Review Reviewed-by Neutral LLM
Reviewer Boris Burkov pointed out that the patch's assumption about extent buffers being eligible for release_folio() after refs == 1 is incorrect, as they should be immediately released when a block gets cowed, and instead suggested that the folios of interest are those with refs == 1 that will return 0 to release_folio(), which can already be reclaimed by memory pressure based reclaim. Reviewer Boris Burkov suggested adding a Reviewed-by tag and left a small code nit, but did not raise any technical concerns about the patch's functionality. Reviewer Boris Burkov suggested that instead of passing a __GFP_MOVABLE boolean flag, the code should transparently pass along a GFP flag to let callers handle movable pages correctly.

Small correction to a generally correct story:

every extent_buffer we have used (it is newly cowed or we have read it
for some reason) will settle into a state where extent_buffer->refs == 1
and it is eligible for release_folio(). If a block gets cowed, then its
extent_buffer should be fully released/freed right away, with a tricky
EXTENT_BUFFER_STALE bit dance to prevent accidental reuse during this
process.

So I don't think cow or trees moving around is all that relevant here.

The folios you are after are the ones which are sitting around with
extent_buffer->refs == 1 and which will return 0 to release_folio(),
which is any valid, non-stale extent_buffer no btrfs operation is
currently using (typically referred to from a btrfs_path). These are
already available to be reclaimed by memory pressure based reclaim,
drop_caches, etc.
