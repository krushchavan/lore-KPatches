Last comment 2026-06-25 [RFC PATCH] mm: bypass swap readahead for zswap Needs Work LLM (per-reviewer) RFC

This patch bypasses swap readahead for zswap, an in-memory compression backend similar to zram. The current implementation of swap readahead is optimized for traditional disk-based swap devices and can cause issues like inflated anon residency, page cache thrashing, and increased fault latency when applied to an in-memory backend. By skipping readahead for zswap, the patch aims to improve performance by reducing unnecessary prefetching of pages that are not needed. The results from testing this bypass show significant improvements in build time, swap-in/out rates, and reduced swap readahead hits.

This patch series adds atomic whole-device hotplug control to the dax kmem driver via a sysfs 'state' attribute, allowing for transitions across entire regions of memory to be performed atomically and safely. The current implementation has a race condition between offlining and removing memory blocks, which this patch aims to solve by introducing a new ABI with state names mirroring per-block memoryX/state attributes. The series also includes mm preparation changes to support the new hotplug control, including adding helpers for online type management and atomic offline+remove of multiple ranges. The dax/kmem feature is updated to plumb online_type through its device creation path and extract hotplug/hotremove into helper functions. A selftest is added to regression test the new attribute.

[PATCH v1] fuse: don't clear folio uptodate on writethrough errors Neutral llm-chunked-no-replies Awaiting Review

This patch prevents the clearing of a folio's uptodate flag in case of write-through errors on FUSE filesystems. This is necessary to maintain the invariant that non-uptodate folios must not be visible to userspace, which could lead to potential information leaks. The change ensures that if an application writes data and the server fails to persist it or only services a short write, the page cache folio will retain the original data instead of reverting to the server's contents on subsequent reads.

🔥 TODAY [PATCH v1] fuse: enable large folios Needs Work LLM (per-reviewer) Under Review

This patch enables large folios for the FUSE file system by capping the maximum order at the largest request that can be issued to a fuse device, ensuring that each folio fits within a single request. This change allows the allocator to fall back to smaller folios under memory pressure. The patch includes benchmark results showing significant performance improvements in various workloads, including sequential reads and writes, with some cases seeing up to 141% increase in throughput.

This patch series moves the per-cpu stock from struct mem_cgroup to page_counter, reducing memory footprint and eliminating locking and synchronous work_on_cpu. The new implementation uses atomic operations for stock management and allows for independent stock management for additional page_counters in each memcg. 🔥 TODAY Re: mm/hwpoison: persist poisoned PFN list across kexec via KHO [RFC] Needs Work LLM (per-reviewer) RFC

This patch addresses a problem where poisoned PFN (Page Frame Number) lists are not persisted across kexec, which can lead to memory failures in the new kernel. The issue arises because some memory allocations occur before buddy is initialized and can land on bad frames. To solve this, the patch proposes using EFI_UNUSABLE_MEMORY to mark non-usable memory as nomap, making it invisible to both allocators and KASLR. This approach ensures that the bad frame is not accessible in the new kernel.
