Ongoing Submitted 2026-03-10 🔥 TODAY [PATCH v2 0/2] locking: contended_release tracepoint instrumentation v1 → v2 Needs Work LLM (per-reviewer) Under Review

This patch adds a new tracepoint called contended_release that fires on the lock holder's side when a lock is released while there are waiters. This provides additional information such as hold time estimation and the holder's stack at release time, which can be useful for debugging and performance analysis. The new tracepoint has minimal impact on performance and binary size. It was previously discussed in an RFC (Request For Comments) and this patch addresses some of the feedback received.

Ongoing Submitted 2026-03-10 🔥 TODAY [PATCH v2 2/2] locking: Add contended_release tracepoint v1 → v2 Needs Work LLM (per-reviewer) Under Review

This patch adds a new tracepoint called contended_release, which fires when a lock is released under contention. The patch complements existing contention_begin and contention_end tracepoints by providing information from the holder's perspective. It enables correlating lock hold time under contention with waiter events by lock address. The patch introduces calls to trace_contended_release() in slowpath unlock paths of various sleepable locks, including mutex, rtmutex, semaphore, rwsem, percpu-rwsem, and RT-specific rwbase locks.
