Ongoing Submitted 2026-04-03 🔥 TODAY [PATCH v2 1/5] btrfs: reserve space for delayed_refs in delalloc v1 → v2 Needs Work LLM (per-reviewer) New Version Expected

This patch addresses a problem where the Btrfs file system does not reserve enough space for delayed references (delayed_refs) during delalloc operations. Delalloc uses a per-inode block reservation to anticipate metadata needs, but it does not account for delayed refs. As a result, when writeback occurs and delayed refs are generated, they draw from an under-reserved trans_handle's delayed_refs_rsv, leading to incorrect assessments of reserved space in the metadata space reclaim logic. The patch fixes this by reserving space for delayed refs during delalloc operations.

This patch addresses an issue where the btrfs kernel module under-reserved space for delayed reference reservations during delalloc writes when compression is enabled. The problem arises from an incorrect estimate of outstanding extents due to a mismatch between maximum uncompressed and compressed extent sizes. To fix this, the patch introduces new functions in btrfs_inode.h to calculate the maximum extent size based on inode properties and uses these values to accurately reserve space for delayed references. As a result, the patch improves the accuracy of delalloc reservation calculations and prevents under-reservation issues even when compression is forced or always enabled for an inode.

Ongoing Submitted 2026-04-03 🔥 TODAY [PATCH v2 4/5] btrfs: make inode->outstanding_extents a u64 v1 → v2 Needs Work LLM (per-reviewer) Under Review

This patch updates the data type of the 'outstanding_extents' field in the Btrfs inode structure from an unsigned integer (u32) to a 64-bit unsigned integer (u64). This change is motivated by the fact that the maximum file size is limited by LLONG_MAX, which is approximately 2^63. As a result, the maximum extent size computation in Btrfs has always been bounded above by this value, making it necessary to use a larger data type to accurately represent this value. The patch also updates several functions and variables to work with the new u64 data type.

Ongoing Submitted 2026-04-03 🔥 TODAY [PATCH v2 5/5] btrfs: cap shrink_delalloc iterations to 128M v1 → v2 Neutral LLM (per-reviewer) Under Review

This patch limits the iterations of the `shrink_delalloc` function in Btrfs to prevent it from blocking indefinitely, especially when dealing with large amounts of writeback data. The function now iterates over delalloc work in units of 128MiB, rather than trying to reclaim all available space at once. This change aims to strike a balance between making progress on delalloc and not starving other types of reservations, which can lead to system stalls and long commit times.

This patch adds a warning to the Linux kernel when a dentry (directory entry) is freed while its 'd_lru' list is not empty. This is done to prevent use-after-free errors, which can cause panics in the system. The problem was identified through a series of panics on hosts with high process churn and was also reported by syzbot. The patch adds a WARN_ON_ONCE() call to the dentry_free function to catch this issue and make it easier to debug.

  • [2/2] mm: page_alloc: per-cpu pageblock buddy allocator 🔥 TODAYNeeds WorkAccepted Submitted 2026-04-03 Rik Riel Johannes Weiner (author) Frank Linden
Needs Work LLM (per-reviewer) Accepted

This patch introduces a new page allocator called pcp buddy allocator, designed to improve scalability and reduce contention in high-thread-count and large-memory systems. The current PCP (Page Cache Pool) has limitations when dealing with concurrent allocations and frees, leading to zone->lock contention. This patch addresses these issues by allowing the PCP to grab entire pageblocks, split them outside the zone->lock, and take ownership of the block. This approach reduces lock contention, increases reuse of pages, and allows for upfront merging of buddies under the local pcp->lock. However, it also introduces concerns about fragmentation, which are mitigated by remembering partial blocks on drain and queuing their pages last on the zone freelist.

🔥 TODAY Re: [RFC 0/2] mm: page_alloc: pcp buddy allocator Needs Work LLM (per-reviewer) Accepted

This patch introduces a per-CPU cache (PCP) buddy allocator for page allocation in the Linux kernel. The PCP is a per-cpu cache of limited size that stores pages to reduce contention on the zone lock. The patch also includes affinity routing rules to increase the chances of successful merges between CPUs. This approach aims to improve performance by reducing lock contention and deferring merging, but it does not change the underlying block economy or buddy code. The impact on alloc_contig should remain the same.
