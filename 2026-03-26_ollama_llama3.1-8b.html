🔥 TODAY [PATCH v4 0/5] locking: contended_release tracepoint instrumentation Needs Work LLM (per-reviewer) Under Review

This patch series adds a new tracepoint called contended_release that fires on the lock holder's side when a lock is released while there are waiters. This provides two main benefits: it allows for hold time estimation by correlating contention_end and contended_release events, and it captures the holder's stack at release time, which can be different from what is captured earlier if the holder does significant work between the waiter's arrival and unlock. The patch also fixes several issues in the previous versions, including spurious events, use-after-free errors, and build failures on certain architectures.

[PATCH v4 3/5] locking: Add contended_release tracepoint to sleepable locks Neutral llm-chunked-no-replies Awaiting Review

This patch adds a new tracepoint called contended_release to the Linux kernel's locking subsystem. The tracepoint fires when a lock is released under contention, allowing developers to correlate lock hold times with waiter events by lock address. The patch includes modifications to various slowpath unlock paths for sleepable locks, including mutexes, rtmutexes, semaphores, rwsems, and percpu-rwsems. For reader/writer locks, the tracepoint fires for every reader releasing while a writer is waiting, not just the last reader.

[PATCH v4 5/5] locking: Add contended_release tracepoint to spinning locks Positive llm-chunked-no-replies Awaiting Review

This patch adds a contended_release tracepoint to queued spinlocks and rwlocks, allowing for tracing of lock contention events. The patch uses an arch-overridable function to ensure correct behavior across all architectures, including those with custom unlock implementations. When the tracepoint is disabled, it has minimal impact on performance, adding only a single NOP instruction to the hot path. However, when enabled, it can increase binary size by up to 0.24% due to the need to preserve the lock pointer in a callee-saved register across the trace call.

Last comment 2026-03-27 [PATCH] mm/shmem: use invalidate_lock to fix hole-punch race Needs Work LLM (per-reviewer) Under Review

This patch fixes a race condition in the shmem (shared memory) module that can cause 'still mapped when deleted' errors and guest kernel panics. The issue occurs when inflating a VM's balloon while a helper process forks and executes, triggering two races: one around fault handling and another due to TOCTOU (time-of-check-to-time-of-use). To fix these issues, the patch introduces the use of invalidate_lock, which is shared under mmap_lock in fault handlers. This approach matches existing patterns used by other modules and requires removing an unnecessary rcu_read_lock from do_fault_around().

🔥 TODAY [PATCH v1] mm: fix writeback regression for strictlimit BDIs Needs Work LLM (per-reviewer) Under Review

This patch fixes a writeback regression caused by the removal of an unconditional writeback kick in balance_dirty_pages(). The issue arises when strictlimit BDIs (e.g., fuse) are used, as their throttling is calculated using per-wb thresholds rather than global thresholds. This leads to two problems: background writeback is never proactively kicked off, and while IO is throttled, the flusher is still not started. The patch fixes this by kicking off the flusher if wb_dirty exceeds wb_bg_thresh for strictlimit BDIs, restoring performance back to its original baseline.

Ongoing Submitted 2026-03-20 🔥 TODAY [PATCH] btrfs: add BTRFS_IOC_GET_CSUMS ioctl Needs Work LLM (per-reviewer) Under Review

This patch adds a new ioctl (input/output control) called BTRFS_IOC_GET_CSUMS, which allows userspace to query the on-disk checksums for a file without needing to read the data itself. The ioctl takes an argument specifying the offset and length of interest, as well as a buffer to store the results. The kernel returns a struct containing information about whether the checksum is available, and if so, what type it is (e.g., sparse, preallocated, or compressed). This feature is primarily intended for speeding up mkfs.btrfs --rootdir operations, particularly when using the --reflink option, which can result in significant performance improvements.
