Last comment 2026-06-16 [PATCH v3] btrfs: release extent lock per folio in readahead v2 → v3 Needs Work LLM (per-reviewer) Accepted

This patch modifies the Btrfs file system's readahead behavior to release extent locks per folio, rather than holding them for the entire range. This change is intended to prevent a race condition between readahead and direct IO writes, which can cause tasks to hang on the extent lock. The problem arises when readahead fails due to memory allocation issues, leaving pages in an inconsistent state and causing subsequent reads to block on the extent lock. By releasing locks per folio, the patch aims to alleviate this issue and reduce stress on the system.

🔥 TODAY Re: [PATCH v2] btrfs: release extent lock per folio in readahead Needs Work LLM (per-reviewer) Accepted

This patch addresses a contention issue on the extent lock in Btrfs, which occurs when multiple tasks are reading an overlapping range of a compressed file and one task is stuck in direct reclaim. 🔥 TODAY [PATCH v3 0/3] arm64: cross-CPU NMI via SDEI Needs Work LLM (per-reviewer) Under Review

This patch series adds a new mechanism to deliver NMI (Non-Maskable Interrupt) across CPUs on arm64 architecture, using the SDEI (System Description Event Infrastructure) protocol. The goal is to enable interrupting a CPU that has its interrupts locally masked, which is necessary for certain debug and observability features such as system backtraces and crash dumps. The existing pseudo-NMI mechanism is too expensive in terms of performance overhead, so this new approach uses SDEI events delivered by firmware to reach the target CPU without modifying the interrupt mask/unmask hot path. This patch series does not add a hard-lockup detector but provides a way for the backtrace and crash-stop paths to actually reach a masked CPU.

[PATCH v3 1/3] firmware: arm_sdei: add SDEI_EVENT_SIGNAL support Neutral llm-chunked-no-replies Awaiting Review

This patch adds a new function called sdei_event_signal() to the Linux kernel's firmware/arm_sdei module. This function signals a software-signalled event (event 0) to another processor element (PE), making it pending and delivering an NMI-like interrupt even if interrupts are masked on that PE. The function is designed to be safe for use in non-maskable interrupt (NMI) or crash contexts, as it does not take any locks. The patch also updates the include files to reflect the new function's existence.

🔥 TODAY [PATCH v3 2/3] drivers/firmware: add SDEI cross-CPU NMI service for arm64 Needs Work LLM (per-reviewer) Under Review

This patch adds a cross-CPU NMI service for arm64 using the SDEI (System Description Event Infrastructure) software-signalled event mechanism. The goal is to deliver an NMI-like event to an interrupt-masked CPU without the overhead of the pseudo-NMI hot-path. This is achieved by registering a handler for event 0 and signaling it with sdei_event_signal(0, mpidr). The first user of this service is arch_trigger_cpumask_backtrace(), which can now reach masked CPUs. If SDEI is absent or not enabled, the patch falls back to the IPI path. This change also adds a new file drivers/firmware/arm_sdei_nmi.c and updates several other files.

🔥 TODAY [PATCH v3 3/3] arm64: escalate smp_send_stop() to an SDEI NMI as a last resort Needs Work LLM (per-reviewer) Under Review

This patch adds a new mechanism to the Linux kernel on arm64 architecture to handle CPUs that are wedged and ignoring interrupts. When a CPU is wedged, it ignores stop IPIs and without pseudo-NMI there's no NMI IPI to escalate to. The patch introduces a third rung in smp_send_stop() that signals an SDEI event 0 on the remaining online CPUs, which firmware delivers regardless of the target's DAIF. This allows the CPU to be stopped even if it's ignoring interrupts. The patch also folds stop bookkeeping into a shared function arm64_nmi_cpu_stop().
