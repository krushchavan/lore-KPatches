Ongoing Submitted 2026-02-27 [PATCH v4 0/5] mm: zone lock tracepoint instrumentation v2 → v3 → v4 Needs Work LLM Under Review

This patch adds tracepoint instrumentation to zone lock acquire and release operations in the Linux kernel. The implementation uses a lightweight inline helper to check if tracing is enabled, and calls an out-of-line helper when tracing is active. When CONFIG_TRACING is disabled, the helpers compile to empty inline stubs, leaving the fast path unaffected. This patch introduces three new tracepoints: zone_lock_start_locking, zone_lock_acquire_returned, and zone_lock_released.

Ongoing Submitted 2026-02-27 [PATCH v4 4/5] mm: rename zone->lock to zone->_lock v3 → v4 Needs Work LLM Under Review

This patch renames the 'lock' field in the Linux kernel's memory management (mm) code to '_lock'. The rename is intended to break direct users of the lock at compile time, forcing them to use wrappers and tracing infrastructure. This change does not introduce any functional changes but rather enforces the use of standardized locking mechanisms. The patch affects various files in the mm directory, including include/linux/mmzone.h, include/linux/mmzone_lock.h, and several C files such as compaction.c and page_alloc.c.

Re: [PATCH 0/8] mm/zswap, zsmalloc: Per-memcg-lruvec zswap accounting Neutral LLM Under Review

This patch series introduces per-memcg-lruvec zswap accounting, which allows the Linux kernel to track and manage compressed memory usage at a more granular level. The goal is to improve memory management and reduce overhead in systems with multiple NUMA nodes. The patches include benchmark results that show minimal regressions (< 1%) and even some improvements in standard deviation for certain benchmarks. The approach taken involves modifying the zswap and zsmalloc code to account for per-memcg-lruvec usage, which should lead to better memory allocation and deallocation decisions.
