🔥 TODAY Re: [PATCH] fuse: disable default bdi strictlimiting Needs Work LLM (per-reviewer) Under Review

This patch disables the default strictlimiting of block device I/O (bdi) for FUSE file systems, which can lead to inaccurate dirty page statistics and excessive writeback. The issue arises from periodic flushing of memory cgroup stats, causing lagging values that result in under-reporting of dirty pages. This can cause writers to exceed the limit by more than 2x before throttling kicks in. The patch aims to address this problem by disabling strictlimiting for FUSE file systems.

Contentious LLM (per-reviewer) Under Review

This patch set fixes a use-after-free (UAF) bug in the Linux kernel's inode handling when splitting a file folio past the end of file (EOF). The bug occurs because the inode is evicted from memory while still being accessed by the split function. To fix this, the patch changes the way memory_failure() splits files: instead of locking and splitting at the tail anchor, it now locks and splits at the head anchor, which remains in the page cache and keeps the inode pinned. Additionally, the patch makes it explicit that a file folio cannot be split when the anchor is beyond EOF, preventing future callers from reintroducing the bug. The changes are tagged for stable and have been compile- and boot-tested.

Last comment 2026-07-16 Re: [PATCH v2 1/5] mm/memory-failure: keep the folio, not the poisoned subpage, locked across split Positive LLM (per-reviewer) Under Review

This patch fixes a bug in the memory failure handling of huge pages by ensuring that the folio, not just its poisoned subpage, remains locked across split operations. The issue arises when the mapping is unlocked after the split, allowing concurrent final iput() to evict and RCU-free the inode before i_mmap_unlock_read() touches the i_mmap_rwsem. To fix this, the patch moves the unlock of i_mmap_rwsem to right after remap_page(), before the loop that unlocks and frees the after-split folios, and clears @mapping to prevent it from being unlocked again.

This patch moves the check for whether a filesystem's nr_cached_objects hook is memcg-aware from fs/super.c to individual filesystems, allowing each filesystem to handle its own memcg awareness independently. The patch introduces a helper function mem_cgroup_shrink_is_root() in <linux/memcontrol.h> to simplify this check. This change fixes an issue where the assumption that non-root memcgs are not aware of nr_cached_objects was incorrect for XFS, which intentionally drives inode reclaim from per-memcg contexts. The patch also removes a gate in fs/super.c and adds support for XFS's inode-reclaim hook to be driven from per-memcg contexts.

Ongoing Submitted 2026-07-14 🔥 TODAY [PATCH] sched/psi: use for_each_set_bit() in psi_group_change() task-count walk Needs Work LLM (per-reviewer) Under Review

This patch improves the efficiency and readability of the psi_group_change() function in the Linux kernel's scheduler by replacing two loops that iterate over bitmasks with more efficient and easier-to-read calls to for_each_set_bit(). The original code would visit every position up to the highest set bit, even if it was sparse, resulting in unnecessary iterations. The new implementation uses a single loop to iterate only over the set bits, reducing the number of iterations and improving performance on hot scheduler workloads.
