🔥 TODAY Re: [PATCH V10 00/10] famfs: port into fuse Needs Work LLM (per-reviewer) Accepted

This patch series ports the famfs file system into fuse, allowing for a generic mapping interface that can be used by multiple servers. The alternative approach involves using bpf maps to find and compute extent mappings, which is minimal and already written. This approach would require renaming functions in fs/fuse/famfs.c to generic names and moving daxdev setup to connection initialization time. The main concerns are whether this approach introduces technical limitations on famfs's production workloads, such as performance drops or feasibility issues with high-scale systems.

🔥 TODAY Re: fuse/io-uring: Proposal to support pBuf in additon to kBuf Neutral LLM (per-reviewer) Accepted

This patch proposes to support pBuf in addition to kBuf for the fuse and io-uring interfaces. The change comes from a revised approach where kernel-managed buffers are no longer an integral part of the io-uring infrastructure, but rather self-contained within the fuse implementation. This design allows for user-allocated buffers to be passed during registration time, with optional pinning for optimization. Zero-copy requires both header and payload buffer pinning, which is subject to CAP_SYS_ADMIN privileges. The patch addresses concerns about mlock requirements and aims to maintain compatibility with existing fuse and io-uring contracts.

Ongoing Submitted 2026-04-04 🔥 TODAY [PATCH v2] mm/percpu, memcontrol: Per-memcg-lruvec percpu accounting Needs Work LLM (per-reviewer) Under Review

This patch modifies the Linux kernel's memory accounting system to provide more detailed information about per-memcg-lruvec (per-cgroup, per-node, and per-LRU vector) percpu allocations. It does this by converting a certain type of memory tracking from a per-memcg level to a per-node level, allowing for better visibility into memory pressure on multi-NUMA systems. The patch also introduces separate accounting for node-level statistics and memcg-lruvec statistics, which requires exposing a new function to be used outside of the memory control module. This change improves the kernel's ability to track memory usage at a finer granularity, particularly for percpu allocations, but has a small memory overhead and negligible performance impact.

Re: [PATCH v2] mm/percpu, memcontrol: Per-memcg-lruvec percpu accounting Neutral LLM (per-reviewer) Under Review

This patch addresses an issue where per-memcg-lruvec percpu accounting is inaccurate due to CPU hotplug and memory pressure causing pcpu allocations to fall back to another node. The problem arises because the allocated memory can be serviced from a different node, making it difficult to charge percpu memory based on the number of CPUs present on a node. The patch attempts to tackle this issue by exploring various approaches, but none have been successful so far.

🔥 TODAY Re: [RFC, PATCH 00/12] userfaultfd: working set tracking for VM guest memory Needs Work LLM (per-reviewer) RFC

This patch introduces working set tracking for VM guest memory using the userfaultfd mechanism, allowing for serialization of reclaim against memory accesses and enabling the decision on where to place cold pages from userspace. The approach involves flipping a single bit under the mmap lock to switch between tracking and reclaiming, ensuring that read-touched pages are not swapped out. This patch is part of a larger series (PATCH 00/12) and aims to improve memory management in VM guests.
