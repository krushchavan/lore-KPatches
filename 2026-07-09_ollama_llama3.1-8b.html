Last comment 2026-07-10 [PATCH] btrfs: write-protect folios during data writeback Needs Work LLM (per-reviewer) Under Review

This patch addresses a problem where, due to a recent change in the Btrfs kernel module, folios (large blocks of data) are not being properly write-protected during data writeback operations. This can lead to corruption and bugs when processes try to modify sectors while they're being used by writeback. The patch reintroduces the necessary write-protection mechanism without dropping the dirty flag on the folio, which would cause other issues. It also adds a debug warning to help prevent similar regressions in the future.

Last comment 2026-07-10 Re: [PATCH] btrfs: write-protect folios during data writeback Neutral LLM (per-reviewer) Under Review

This patch introduces a new function called lock_delalloc_folios() that write-protects folios during data writeback in Btrfs. The goal is to ensure that multiple zoned folios submitted by run_delalloc_range are properly cleaned, which was previously not guaranteed due to the way extent_writepage and other functions handled dirty pages. By locking these folios, the patch prevents them from being modified or accessed until they are fully written back, thereby maintaining data integrity.

Needs Work LLM (per-reviewer) Under Review

This patch series adds atomic whole-device hotplug control via sysfs, allowing the state of an entire region to be controlled in a single operation. The current implementation onlines memory during probe using the system default policy, but this can lead to races between offlining and removing a whole region. The new feature provides a 'state' attribute that mirrors the per-block memoryX/state ABI, with states for unplugged, online, online_kernel, and online_movable. This allows for atomic transitions across every range of the device, solving the problem of interfering userland controllers during offline and remove steps.

This patch series adds a new mechanism to deliver interrupts across CPUs on arm64 systems, specifically targeting scenarios where the CPU has its interrupts locally masked. Ongoing Submitted 2026-06-29 ⏱ No comments yet [PATCH v5 4/4] arm64: escalate smp_send_stop() to an SDEI NMI as a last resort Neutral llm-chunked-no-replies Awaiting Review

This patch adds a new mechanism to the Linux kernel on arm64 architecture to handle CPUs that are wedged and ignoring interrupts. When a CPU is wedged, it ignores stop IPIs and without pseudo-NMI, there's no NMI IPI to escalate to. The patch introduces a third rung in smp_send_stop() that signals an SDEI event on the remaining online CPUs, which firmware delivers regardless of the target's DAIF. This allows the CPU to be shut down even if it's ignoring interrupts. The patch also folds stop bookkeeping into a shared function arm64_nmi_cpu_stop(), replacing near-duplicate local functions.

Needs Work LLM (per-reviewer) Under Review

This patch series adds support to userfaultfd (a Linux kernel interface) for tracking the working set of virtual machine guest memory, allowing a Virtual Machine Manager (VMM) to identify hot pages and reclaim cold ones to slower storage. The approach taken is to introduce a new registration mode, UFFDIO_REGISTER_MODE_RWP, which uses the same mechanism as existing modes but with additional functionality for working set tracking. This enables the VMM to detect which pages are still being accessed, safely reclaim cold pages, and fetch them back on demand when needed. The patch series includes 15 patches that address various issues and conflicts, including rebasing onto mm-new, resolving a conflict with a hugetlb swap-corruption fix, and tolerating EINVAL from madvise(MADV_NOHUGEPAGE). The patches have been tested and show no regressions in the selftests/mm/uffd-unit-tests.
