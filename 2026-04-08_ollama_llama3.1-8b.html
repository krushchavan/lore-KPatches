Ongoing Submitted 2026-04-07 🔥 TODAY [PATCH v3 0/4] btrfs: improve stalls under sudden writeback v1 → v2 → v3 Needs Work LLM (per-reviewer) Under Review

This patch series aims to improve stalls under sudden writeback in Btrfs file systems, particularly on large memory systems with high dirty ratios. The problem arises when a system accumulates hundreds of gigabytes of dirty pages before triggering writeback, leading to a tailspin of metadata space management and eventual system stall. To address this issue, the patches introduce improved reservations for delayed refs in delalloc work and basic smoothing of flush_space() heuristics. This is achieved through targeted changes to Btrfs inode accounting, types counting, and shrink_delalloc iteration capping.

Ongoing Submitted 2026-04-07 🔥 TODAY [PATCH v3 1/4] btrfs: reserve space for delayed_refs in delalloc v1 → v2 → v3 Needs Work LLM (per-reviewer) Under Review

This patch addresses a problem in the Btrfs file system where space is not reserved for delayed references (delayed_refs) during delalloc operations, leading to incorrect assessments of available metadata space and poor decision making in reclaim logic. The patch adds a reservation for delayed_refs to the per-inode block_rsv, ensuring that enough space is allocated for these references. This fix aims to prevent issues such as erroneous preemptive reclaims and blocking start_transaction() calls due to insufficient reserved space.

Ongoing Submitted 2026-04-07 🔥 TODAY [PATCH v3 2/4] btrfs: account for compression in delalloc extent reservation v1 → v2 → v3 Neutral LLM (per-reviewer) Under Review

This patch addresses an issue in the Btrfs file system where delayed reference reservations were under-estimated due to compression. When compression is enabled, the maximum extent size is reduced from 128MiB to 128KiB, leading to a significant discrepancy between the estimated and actual number of extents required for delalloc writes. The patch introduces a new function `btrfs_inode_max_extents` that takes into account the compressed extent size when calculating the number of outstanding extents. This change ensures accurate reservation of delayed reference space, even with compression enabled. Additionally, the patch simplifies the code by removing an unnecessary shim function and reducing the complexity of the calculation.

Ongoing Submitted 2026-04-07 🔥 TODAY [PATCH v3 3/4] btrfs: make inode->outstanding_extents a u64 v1 → v2 → v3 Neutral LLM (per-reviewer) Under Review

This patch updates the type of the 'outstanding_extents' field in the Btrfs inode structure from a u32 to a u64. This change is motivated by the fact that the maximum file size is LLONG_MAX, which would cause the max extent size computation to exceed the range of a u32 even when considering compressed cases. The patch also updates the 'mod_outstanding_extents' function to handle 64-bit arithmetic and adds assertions to ensure that the inputs are within valid ranges.

This change aims to strike a balance between making progress on space reclaim and avoiding arbitrarily long syscall latencies and task hangs.

Ongoing Submitted 2026-04-06 🔥 TODAY [PATCH] dcache: warn when a dentry is freed with a non-empty ->d_lru Needs Work LLM (per-reviewer) Changes Requested

This patch adds a warning when a dentry is freed while its ->d_lru list is not empty, which can indicate an use-after-free (UAF) bug. The problem was observed in hosts with heavy process churn and resulted in panics during /proc entry invalidation. The patch aims to make it easier to spot the underlying issue by adding a WARN_ON_ONCE() condition when freeing a dentry with a non-empty ->d_lru list.
