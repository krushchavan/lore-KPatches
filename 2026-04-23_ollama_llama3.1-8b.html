🔥 TODAY [PATCH RFC] sunrpc: hardcode pool_mode to pernode, remove other modes Needs Work LLM (per-reviewer) Under Review

This patch removes the SVC_POOL_* enum and mode selection heuristic, simplifying the pool map functions to always use the pernode path. The percpu mode is removed as it created excessive pools relative to the number of threads most deployments run. The module parameter and netlink interfaces are preserved for backward compatibility, but writing any value other than 'pernode' will return an error. The default pool mode is hardcoded to pernode, which provides one pool per node with proper thread affinity and NUMA-local memory allocation.

Last comment 2026-04-24 [PATCH] mm/lruvec: preemptively free dead folios during lru_add drain Needs Work LLM (per-reviewer) Under Review

This patch optimizes the Linux kernel's memory management by preemptively freeing dead folios during the lru_add process, which reduces lock contention and improves system performance. The patch identifies that a significant portion of observable lruvec lock contention occurs when dead folios are present in lru_add batches at drain time, leading to unnecessary lock acquisitions. To address this issue, the patch uses folio_ref_freeze() to filter out dead folios before they enter the LRU and moves them onto a temporary batch for deferred cleanup. The patch was tested on a production workload and showed significant improvements in system-wide memory stats, including reduced direct reclaim scanning, allocation stalls, compaction stalls, and page frees.

Last comment 2026-04-24 [PATCH 0/2] btrfs: skip COW for written extent buffers Needs Work LLM (per-reviewer) Under Review

This patch series optimizes Btrfs by skipping redundant copy-on-write (COW) operations on extent buffers that have already been written back to disk due to memory pressure. The optimization relies on the WRITTEN flag set by writeback, which indicates that a buffer has been committed to disk and can be safely overwritten in place. This reduces COW amplification, which can exhaust block reservations and degrade throughput under memory pressure. The patches also fix a latent issue in qgroup snapshot accounting, where root items were being persisted mid-transaction, leading to incorrect state and potential corruption. The skip-COW optimization is applied when should_cow_block() encounters a WRITTEN buffer from the current transaction, re-dirtying the buffer instead of allocating a new block.

Last comment 2026-04-24 [PATCH 1/2] btrfs: don't finalize fs roots during qgroup snapshot accounting Needs Work LLM (per-reviewer) Under Review

This patch modifies the Btrfs file system to avoid finalizing file system roots during qgroup snapshot accounting, which can lead to incorrect state and silent corruption of snapshots. The patch introduces a new helper function `prepare_root_item` that updates root items without the side effects of finalization, and a new function `qgroup_commit_fs_roots` that iterates over dirty roots using an advancing index instead of clearing tags. This change prevents premature clearing of `BTRFS_ROOT_FORCE_COW`, freeing of log trees, and other finalization actions during transaction progress, ensuring consistent qgroup accounting and preventing potential corruption issues.

[PATCH 2/2] btrfs: skip COW for written extent buffers allocated in current transaction Neutral llm-chunked-no-replies Awaiting Review

This patch optimizes Btrfs by skipping unnecessary copy-on-write (COW) operations when a buffer has been written in the current transaction. When memory pressure causes writeback of a recently COW'd buffer, Btrfs sets a flag indicating it's been written. Subsequent searches then re-COW the buffer unnecessarily, leading to performance degradation and potential exhaustion of block reservations. The patch addresses this issue by skipping COW for buffers allocated in the current transaction when they have the WRITTEN flag set and their generation matches the current transaction. This is safe because overwriting in place is crash-safe due to the committed superblock not referencing buffers allocated in the current transaction.

Ongoing Submitted 2026-04-22 🔥 TODAY [PATCH] btrfs: don't force DIO writes to be serialized Needs Work LLM (per-reviewer) Under Review

This patch addresses an issue in the Btrfs file system where Direct I/O (DIO) writes are being serialized due to a missing flag that was set before the transition to the new mount API in 2023. The flag, SB_NOSEC, tells the VFS that the filesystem may have files without security xattrs, allowing optimizations. Without this flag, DIO writes to the same file will be locked exclusively, leading to reduced performance. This patch removes the serialization by not forcing DIO writes to be exclusive locks, resulting in a significant improvement in DIO throughput, as demonstrated by fio benchmark results showing a 59% increase.
