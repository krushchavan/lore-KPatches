[PATCH v3 1/4] btrfs: reserve space for delayed_refs in delalloc Neutral llm-chunked-no-replies Awaiting Review

This patch addresses an issue in the Btrfs file system where space is not reserved for delayed references (delayed_refs) during delalloc operations. Delalloc uses a per-inode block reservation to anticipate metadata needs, but it does not account for delayed refs. As a result, when writeback occurs and delayed refs are generated, they draw from an under-reserved trans_handle's delayed refs reservation. This can lead to incorrect assessments of reserved space in the metadata space reclaim logic, causing poor decision making and blocking file system operations.

[PATCH v3 2/4] btrfs: account for compression in delalloc extent reservation Neutral llm-chunked-no-replies Awaiting Review

This patch addresses an issue in the Btrfs file system where delayed reference reservations were under-estimated due to compression. Specifically, when COMPRESS_FORCE is set or the inode is configured to always compress, the maximum compressed extent size (128KiB) is significantly smaller than the maximum uncompressed extent size (128MiB). The patch introduces a new function `btrfs_inode_max_extents` that takes into account the compression type and returns the correct number of extents for reservation. This change prevents under-reservation of delayed refs reservations for delalloc writes, even when compression is forced. Additionally, the patch removes an unnecessary shim function `count_max_extents` and simplifies the code in the process.

[PATCH v3 3/4] btrfs: make inode->outstanding_extents a u64 Neutral llm-chunked-no-replies Awaiting Review

This patch updates the data type of 'outstanding_extents' in the Btrfs inode structure from u32 to u64. This change is motivated by the fact that the maximum file size is LLONG_MAX, which results in a computation bound above by ~2^63 / 2^27. With recent changes to divide by 128KiB in compressed cases, this bound is even higher. The patch also updates the 'mod_outstanding_extents' function to handle u64 inputs and adds assertions to ensure that the inputs are valid. This change aims to capture the intent of the types and make the code more robust.

Ongoing Submitted 2026-03-26 🔥 TODAY [PATCH v4 0/5] locking: contended_release tracepoint instrumentation Needs Work LLM (per-reviewer) Under Review

This patch series adds a new tracepoint called contended_release that fires on the lock holder's side when a lock is released while there are waiters. This provides two main benefits: it allows for hold time estimation by correlating contention_end and contended_release events, and it captures the holder's stack at release time, which can be different from what is captured earlier if the holder does significant work between the waiter's arrival and unlock. The patch also fixes several issues in the previous versions, including spurious events, use-after-free errors, and build failures on certain architectures.

The patch introduces an order-aware parameter that allows vmpressure() to differentiate between costly and non-costly orders, asserting socket pressure only for orders above PAGE_ALLOC_COSTLY_ORDER. This change aims to prevent unnecessary throttling of TCP throughput due to poor reclaim efficiency at high orders, while maintaining the existing behavior for memcg reclaim and low-priority reclaim scenarios.

Last comment 2026-04-08 Re: [PATCH] btrfs: add BTRFS_IOC_GET_CSUMS ioctl Needs Work LLM (per-reviewer) Under Review

This patch adds a new ioctl (input/output control) called BTRFS_IOC_GET_CSUMS to the btrfs file system. The ioctl allows userspace to query the checksums of data that would be read from a file at a specific offset, without requiring root privileges or knowledge of the internal workings of btrfs. This is achieved by providing a simpler alternative to existing methods for obtaining checksums, such as using FIEMAP and the tree search ioctl, which are more complex and require administrative capabilities.
